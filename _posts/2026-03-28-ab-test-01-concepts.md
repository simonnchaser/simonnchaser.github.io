A/B 테스트는 기존 버전과 새로운 버전을 실제 사용자에게 무작위로 나눠 보여준 뒤, 어떤 버전이 더 좋은 성과를 내는지 비교하는 실험 방법이다. 흔히 `split test`, `bucket test`라고도 부르며, 온라인 제품 환경에 맞게 적용한 무작위 대조 실험이라고 볼 수 있다.

보통은 아래처럼 구성된다.

- `control`: 현재 운영 중인 기존 버전
- `test`: 새로 시도해보는 버전

예를 들어 현재 결제 페이지를 A로 두고, 단계를 줄인 새 결제 페이지를 B로 두었다면 일부 사용자는 A를, 다른 일부 사용자는 B를 보게 된다. 이후 결제 완료율, 클릭률, 가입률 같은 지표를 비교해 어느 쪽이 더 나은지 판단한다.

## 1. 왜 A/B 테스트를 하는가

A/B 테스트의 목적은 감이 아니라 데이터로 의사결정하기 위해서다.

서비스를 운영하다 보면 같은 문제를 두고도 의견이 갈릴 때가 많다.

- 디자이너는 버튼 색을 바꾸면 클릭률이 오를 거라고 생각할 수 있다.
- PM은 문구를 바꾸는 것이 더 효과적이라고 볼 수 있다.
- 개발자는 큰 차이가 없을 수도 있다고 판단할 수 있다.

이럴 때 말로만 결정하면 주관이 섞이기 쉽다. A/B 테스트를 하면 실제 사용자 반응을 바탕으로 객관적으로 비교할 수 있다.

또 하나의 중요한 이유는 위험을 줄이기 위해서다. 설문조사나 인터뷰에서 반응이 좋아도 실제 서비스 환경에서 행동이 어떻게 나타날지는 다를 수 있다. 그래서 새 기능을 전 사용자에게 한 번에 배포하는 대신 일부 사용자에게만 먼저 노출해보는 방식이 안전하다.

예:

- 5% 사용자에게 먼저 노출
- 문제 없으면 20%
- 성과가 괜찮으면 50%
- 이후 전체 배포

즉, A/B 테스트는 성과 측정 도구이면서 동시에 배포 리스크를 줄이는 운영 방식이기도 하다.

## 2. 핵심 개념은 가설이다

가설 없는 A/B 테스트는 사실상 의미가 약하다.

A/B 테스트는 단순히 이것저것 바꿔보는 작업이 아니라, 어떤 변화를 주면 어떤 지표가 어떻게 달라질 것인지를 검증하는 실험이다. 좋은 가설은 보통 아래 세 가지를 포함한다.

- 무엇을 바꿀 것인가
- 어떤 지표가 변할 것인가
- 어느 방향으로 얼마나 변할 것으로 예상하는가

예:

- "새로운 추천 알고리즘을 적용하면 클릭률이 기존 대비 5% 증가할 것이다."
- "체크아웃 단계를 4단계에서 2단계로 줄이면 결제 완료율이 증가할 것이다."
- "회원가입 폼의 입력 항목을 줄이면 가입 전환율이 올라갈 것이다."

가설이 있어야 실험 결과를 해석할 수 있다. 결과가 좋아도 왜 좋아졌는지 설명할 수 있고, 결과가 나빠도 무엇을 배웠는지 남길 수 있다.

## 3. A/B 테스트는 비교 실험에 잘 맞는다

A/B 테스트는 기존안과 새안을 비교할 수 있는 상황에서 특히 유용하다.

예:

- 현재 추천 방식 vs 새로운 추천 방식
- 현재 버튼 문구 vs 새로운 버튼 문구
- 현재 체크아웃 흐름 vs 간소화된 체크아웃 흐름

즉, 기준점이 있어야 한다. 비교 기준이 명확할수록 실험 결과도 해석하기 쉽다.

## 4. 실제 사례로 보면 더 쉽다

### 추천 시스템 개선

가설:

- "개인화 추천을 적용하면 상품 클릭률과 구매율이 증가할 것이다."

설계:

- A: 현재 인기순 추천
- B: 개인화 추천

측정:

- 추천 영역 클릭률
- 상품 상세페이지 진입률
- 구매 전환율

### 체크아웃 단계 축소

가설:

- "결제 단계를 4단계에서 2단계로 줄이면 결제 완료율이 증가할 것이다."

설계:

- A: 현재 4단계 결제 흐름
- B: 2단계 결제 흐름

측정:

- 결제 완료율
- 이탈률
- 평균 주문 금액

결제 완료율만 올라가고 객단가는 떨어질 수도 있기 때문에 핵심 지표 외에 보조 지표도 함께 봐야 한다.

### 버튼 문구 변경

가설:

- "`지금 구매하기`보다 `오늘 바로 시작하기` 문구가 클릭률을 더 높일 것이다."

설계:

- A: `지금 구매하기`
- B: `오늘 바로 시작하기`

측정:

- 버튼 클릭률
- 결제 전환율

이런 식으로 작고 비교 가능한 변화는 A/B 테스트에 잘 맞는다.

## 5. 여러 실험을 동시에 돌릴 수 있는가

서비스의 서로 다른 영역에 대한 테스트는 동시에 여러 개 실행하기도 한다.

예:

- 홈 화면 배너 문구 테스트
- 검색 결과 정렬 테스트
- 결제 버튼 디자인 테스트

다만 완전히 독립적이라고 가정하면 안 된다. 실험끼리 상호작용이 생길 수 있다.

예를 들어 홈 화면 할인 배너와 상세페이지 구매 버튼 테스트가 동시에 돌아가면, 구매율 변화가 어느 테스트의 영향인지 분리하기 어려울 수 있다.

따라서 여러 실험을 동시에 돌릴 때는 아래를 함께 봐야 한다.

- 같은 핵심 지표에 영향을 주는지
- 같은 사용자 행동 흐름 안에 있는지
- 서로 간섭할 가능성이 있는지

## 6. A/B 테스트를 써야 하는 경우

아래 조건이 어느 정도 충족되면 A/B 테스트를 고려할 수 있다.

- 기존 버전과 새 버전을 비교할 수 있다.
- 측정할 지표가 명확하다.
- 실험할 만큼 트래픽이 있다.
- 명확한 가설이 있다.
- 실제 사용자 반응을 통해 검증할 필요가 있다.

대표 예시:

- 버튼 문구 변경
- 배너 디자인 변경
- 추천 알고리즘 비교
- 검색 정렬 방식 비교
- 회원가입/결제 단계 개선

## 7. A/B 테스트를 쓰지 않는 편이 나은 경우

### 데이터가 너무 적을 때

`No Data, No A/B Test`

트래픽이나 전환 수가 너무 적으면 실험 결과가 우연에 크게 흔들린다.

예:

- 하루 방문자 수가 매우 적은 서비스
- 결제 건수가 거의 없는 페이지

이 경우는 A/B 테스트보다 먼저 아래를 고려하는 편이 낫다.

- 더 긴 기간 데이터 수집
- 상위 퍼널 지표 사용
- 사용자 인터뷰
- 설문조사

여기서 상위 퍼널 지표를 본다는 것은 최종 구매처럼 데이터가 너무 적은 지표 대신, 클릭이나 상세 진입처럼 더 앞단에서 더 자주 발생하는 지표를 먼저 보자는 뜻이다.

### 버그 수정일 때

버그는 실험할 것이 아니라 빠르게 고쳐야 한다.

예:

- 결제 버튼이 특정 브라우저에서 눌리지 않음
- 로그인 오류가 발생함
- 장바구니 계산이 잘못됨

이런 문제는 A/B 테스트로 임팩트를 재기보다 즉시 수정하는 것이 우선이다.

### 아이디어가 아직 너무 추상적일 때

A/B 테스트는 실제 트래픽과 운영 비용을 사용한다. 그래서 아이디어가 아직 구체적이지 않으면 먼저 더 저렴한 검증이 필요하다.

예:

- "우리 서비스에 커뮤니티 기능이 있으면 좋지 않을까?"
- "AI 기능을 넣으면 멋질 것 같은데?"

이런 경우에는 아래 방법이 더 적절할 수 있다.

- 사용자 인터뷰
- 설문조사
- 프로토타입 테스트
- Fake door test

### 가설 없이 랜덤하게 바꿔보는 경우

예:

- "배너 색을 그냥 바꿔보자"
- "문구를 그냥 짧게 해보자"

이런 방식은 실험 후에도 배움이 잘 남지 않는다. 왜 이 변화가 필요한지, 어떤 지표가 바뀔 것인지 설명할 수 있어야 한다.

### 비교 대상이 없는 완전히 새로운 기능일 때

예:

- 서비스에 처음 도입하는 완전히 새로운 소셜 기능
- 기존 흐름과 전혀 다른 구독 모델

이런 경우는 A/B 테스트보다 아래가 더 적합할 수 있다.

- 소규모 베타 테스트
- 프로토타입 검증
- 수요 확인용 Fake door test
- 사용자 리서치

## 8. Fake Door Test란

Fake door test는 실제 기능을 완전히 만들기 전에, 마치 기능이 있는 것처럼 버튼이나 메뉴를 먼저 만들어 사용자 반응을 보는 방법이다.

예:

- 홈 화면에 `AI 추천 받기` 버튼을 넣는다.
- 사용자가 누르면 "준비 중입니다" 페이지로 이동시킨다.
- 클릭률을 보고 실제 수요가 있는지 판단한다.

즉, 기능 개발 전에 관심도부터 확인하는 방식이다.

## 9. 왜 애자일 A/B 테스트가 중요한가

실험 하나를 몇 달씩 끌면 학습 속도와 제품 개선 속도가 같이 느려진다. A/B 테스트는 한 번으로 끝나는 경우가 드물고, 보통 여러 번의 반복을 통해 조금씩 개선 방향을 찾아간다.

예:

- 1차 실험: 버튼 문구 변경
- 2차 실험: 버튼 위치 변경
- 3차 실험: 결제 단계 축소
- 4차 실험: 추천 모델 교체

실험 설정, 로그 확인, 분석, 의사결정이 너무 오래 걸리면 의미 있는 개선까지의 시간도 길어진다. 반대로 실험 시간이 줄어들수록 학습 속도가 빨라지고, 학습 속도가 빨라질수록 제품 출시와 개선도 빨라진다.

특히 ML 기반 A/B 테스트에서는 아래 자동화가 더 중요하다.

- 실험 설정 자동화
- 실험 로그 수집 자동화
- 분석 자동화
- 모델 배포 자동화

즉, A/B 테스트의 경쟁력은 통계 지식만이 아니라 실험을 얼마나 빠르게 반복할 수 있는지에도 달려 있다.

## 10. 짧게 정리

- A/B 테스트는 실제 사용자 반응을 바탕으로 의사결정하기 위한 비교 실험이다.
- 반드시 가설이 먼저 있어야 하고, 무엇을 바꾸는지와 어떤 지표를 볼지 명확해야 한다.
- 기존안과 새안을 비교할 수 있고 충분한 트래픽이 있을 때 특히 유용하다.
- 데이터가 너무 적거나, 버그 수정이거나, 아이디어가 너무 추상적이면 다른 검증 방식이 더 적합할 수 있다.
- 실험은 한 번보다 반복이 중요하므로, 설정과 분석이 빠른 애자일한 운영 체계가 필요하다.
