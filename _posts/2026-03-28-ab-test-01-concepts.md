# A/B Test 01강 개념 정리

## 1. A/B Test란 무엇인가

A/B 테스트는 기존 버전과 새로운 버전을 실제 사용자에게 무작위로 나눠 보여준 뒤,  
어떤 버전이 더 좋은 성과를 내는지 비교하는 실험 방법이다.

다른 말로는 아래처럼도 부른다.

- Split Test
- Bucket Test

오프라인의 무작위 대조 실험(Randomized Controlled Trial)을 온라인 서비스에 적용한 형태라고 볼 수 있다.

보통 다음처럼 구성된다.

- `control`: 현재 운영 중인 기존 버전
- `test`: 새로 시도해보는 버전

필요하면 테스트 버전이 하나가 아니라 여러 개일 수도 있다.

예:

- A안: 현재 결제 페이지
- B안: 결제 단계를 줄인 새 페이지

사용자 일부는 A를 보고, 다른 일부는 B를 본다.  
그 뒤 결제 완료율, 클릭률, 가입률 같은 지표를 비교해 어느 쪽이 더 나은지 판단한다.

## 2. 왜 A/B Test를 하는가

A/B 테스트의 목적은 "감"이 아니라 "데이터"로 의사결정하기 위해서다.

서비스를 운영하다 보면 이런 상황이 자주 생긴다.

- 디자이너는 버튼 색을 바꾸면 클릭률이 오를 거라고 생각한다.
- PM은 문구를 바꾸는 것이 더 효과적이라고 생각한다.
- 개발자는 큰 차이가 없을 수도 있다고 본다.

이럴 때 말로만 판단하면 주관이 섞이기 쉽다.  
A/B 테스트를 하면 실제 사용자 반응을 바탕으로 객관적으로 비교할 수 있다.

즉, A/B 테스트는 다음과 같은 장점이 있다.

- 새로운 기능이나 변경 사항을 객관적으로 측정할 수 있다.
- 전체 사용자에게 한 번에 배포하지 않고 일부 트래픽으로 먼저 검증할 수 있다.
- 큰 위험 없이 빠르게 배우고 개선할 수 있다.

## 3. 핵심 개념: 가설이 먼저다

가설 없는 A/B 테스트는 사실상 의미가 없다.

A/B 테스트는 단순히 "이것저것 바꿔보자"가 아니라,  
"어떤 변화를 주면 어떤 지표가 어떻게 달라질 것인가"를 검증하는 실험이다.

좋은 가설은 보통 아래 세 가지를 포함한다.

- 무엇을 바꿀 것인가
  예: 체크아웃 페이지의 결제 단계를 4단계에서 2단계로 줄인다.
- 어떤 지표가 변할 것인가
  예: 결제 완료율, 결제 단계 이탈률이 변할 것으로 본다.
- 어느 방향으로 얼마나 변할 것으로 예상하는가
  예: 결제 완료율은 3~5% 증가하고, 중간 이탈률은 감소할 것으로 예상한다.

예시 1:

- "새로운 추천 알고리즘을 적용하면 클릭률이 기존 대비 5% 증가할 것이다."

예시 2:

- "체크아웃 페이지의 단계를 줄이면 결제 완료율이 증가할 것이다."

예시 3:

- "회원가입 폼의 입력 항목을 줄이면 가입 전환율이 올라갈 것이다."

가설이 있어야 실험 결과를 해석할 수 있다.  
결과가 좋아도 왜 좋아졌는지 설명할 수 있고, 결과가 나빠도 무엇을 배웠는지 남길 수 있다.

## 4. A/B Test는 보통 비교 실험이다

A/B 테스트는 "기존안 vs 새안"처럼 비교 대상이 있는 상황에 잘 맞는다.

예를 들면:

- 현재 추천 방식 vs 새로운 추천 방식
- 현재 버튼 문구 vs 새로운 버튼 문구
- 현재 체크아웃 흐름 vs 간소화된 체크아웃 흐름

즉, 기준점이 있어야 한다.  
비교 기준이 명확할수록 실험 결과도 해석하기 쉽다.

## 5. 실제 사례로 이해하기

### 사례 1. 추천 시스템 개선

가설:

- "개인화 추천을 적용하면 상품 클릭률과 구매율이 증가할 것이다."

설계:

- A: 현재 인기순 추천
- B: 개인화 추천

측정:

- 추천 영역 클릭률
- 상품 상세페이지 진입률
- 구매 전환율

이 실험을 통해 "개인화 추천이 정말 비즈니스 성과에 도움이 되는지"를 확인할 수 있다.

### 사례 2. 체크아웃 단계 축소

가설:

- "결제 단계를 4단계에서 2단계로 줄이면 결제 완료율이 증가할 것이다."

설계:

- A: 현재 4단계 결제 흐름
- B: 2단계 결제 흐름

측정:

- 결제 완료율
- 이탈률
- 평균 주문 금액

주의할 점은 결제 완료율만 올라가고 객단가는 떨어질 수도 있다는 점이다.  
그래서 핵심 지표 외에 보조 지표도 함께 봐야 한다.

### 사례 3. 버튼 문구 변경

가설:

- "`지금 구매하기`보다 `오늘 바로 시작하기` 문구가 클릭률을 더 높일 것이다."

설계:

- A: `지금 구매하기`
- B: `오늘 바로 시작하기`

측정:

- 버튼 클릭률
- 결제 전환율

이런 식으로 작고 비교 가능한 변화는 A/B 테스트에 잘 맞는다.

## 6. 동시에 여러 A/B Test를 돌릴 수 있는가

보통 서비스의 서로 다른 영역에 대한 테스트는 동시에 여러 개 실행하기도 한다.

예:

- 홈 화면 배너 문구 테스트
- 검색 결과 정렬 테스트
- 결제 버튼 디자인 테스트

다만 완전히 독립적이라고 가정하면 안 된다.  
테스트끼리 상호작용이 생길 수 있다.

예를 들어 홈 화면 할인 배너와 상세페이지 구매 버튼 테스트가 동시에 돌아가면,  
구매율 변화가 어느 테스트 영향인지 분리하기 어려울 수 있다.

따라서 여러 실험을 동시에 돌릴 때는:

- 같은 핵심 지표에 영향을 주는지
- 같은 사용자 행동 흐름 안에 있는지
- 서로 간섭할 가능성이 있는지

를 함께 고려해야 한다.

## 7. A/B Test를 사용해야 하는 경우

아래 조건이 어느 정도 충족되면 A/B 테스트를 고려할 수 있다.

- 기존 버전과 새 버전을 비교할 수 있다.
- 측정할 지표가 명확하다.
- 실험할 만큼 트래픽이 있다.
- 명확한 가설이 있다.
- 실제 사용자 반응을 통해 검증할 필요가 있다.

대표 예시:

- 버튼 문구 변경
- 배너 디자인 변경
- 추천 알고리즘 비교
- 검색 정렬 방식 비교
- 회원가입/결제 단계 개선

## 8. A/B Test를 사용하면 안 되는 경우

### 8-1. 데이터가 너무 없을 때

`No Data, No A/B Test`

트래픽이나 전환 수가 너무 적으면 실험 결과가 우연에 크게 흔들린다.

예:

- 하루 방문자 수가 매우 적은 서비스
- 결제 건수가 거의 없는 페이지

이 경우는 A/B 테스트보다 먼저:

- 더 긴 기간 데이터 수집
- 상위 퍼널 지표 사용
- 사용자 인터뷰
- 설문조사

여기서 상위 퍼널 지표를 본다는 것은  
최종 구매처럼 데이터가 너무 적은 지표 대신, 클릭이나 상세 진입처럼 더 앞단에서 더 많이 발생하는 지표를 먼저 보자는 뜻이다.

같은 방법을 고려하는 편이 낫다.

### 8-2. 버그 수정일 때

버그는 실험할 것이 아니라 빠르게 고쳐야 한다.

예:

- 결제 버튼이 특정 브라우저에서 눌리지 않음
- 로그인 오류가 발생함
- 장바구니 계산이 잘못됨

이런 문제는 A/B 테스트로 임팩트를 재기보다 즉시 수정하는 것이 우선이다.

### 8-3. 아이디어가 아직 너무 추상적일 때

A/B 테스트는 실제 트래픽과 운영 비용을 사용한다.  
그래서 아이디어가 아직 구체적이지 않으면 먼저 더 저렴한 검증이 필요하다.

예:

- "우리 서비스에 커뮤니티 기능이 있으면 좋지 않을까?"
- "AI 기능을 넣으면 멋질 것 같은데?"

이런 경우에는 먼저 아래 방법이 더 적절할 수 있다.

- 사용자 인터뷰
- 설문조사
- 프로토타입 테스트
- Fake door test

### 8-4. 가설 없이 랜덤하게 바꿔보는 경우

예:

- "배너 색을 그냥 바꿔보자"
- "문구를 그냥 짧게 해보자"

이런 방식은 실험 후에도 배움이 잘 남지 않는다.  
왜 이 변화가 필요한지, 어떤 지표가 바뀔 것인지 설명할 수 있어야 한다.

### 8-5. 비교 대상이 없는 완전히 새로운 기능일 때

예:

- 서비스에 처음 도입하는 완전히 새로운 소셜 기능
- 기존 흐름과 전혀 다른 구독 모델

이런 경우는 A/B 테스트보다:

- 소규모 베타 테스트
- 프로토타입 검증
- 수요 확인용 Fake door test
- 사용자 리서치

가 더 적합할 수 있다.

## 9. Fake Door Test란

Fake door test는 실제 기능을 완전히 만들기 전에,  
마치 기능이 있는 것처럼 버튼이나 메뉴를 먼저 만들어 사용자 반응을 보는 방법이다.

예:

- 홈 화면에 `AI 추천 받기` 버튼을 넣는다.
- 사용자가 누르면 "준비 중입니다" 페이지로 이동시킨다.
- 클릭률을 보고 실제 수요가 있는지 판단한다.

즉, 기능 개발 전에 관심도부터 확인하는 방식이다.

## 10. 한 줄 요약

A/B 테스트는 기존안과 새안을 실제 사용자에게 무작위로 나눠 보여주고,  
명확한 가설과 지표를 바탕으로 어떤 변화가 더 좋은지 검증하는 실험 방법이다.

## 11. 꼭 기억할 포인트

- A/B 테스트는 감이 아니라 데이터로 비교하는 방법이다.
- 반드시 가설이 먼저 있어야 한다.
- 비교 가능한 기존 버전이 있을 때 특히 유용하다.
- 데이터가 너무 적거나, 버그 수정이거나, 아이디어가 너무 추상적이면 적합하지 않다.
- 모든 실험은 학습을 남겨야 한다.

## 12. 왜 A/B Test를 하는가

A/B 테스트를 하는 가장 큰 이유는,  
비즈니스 관련 지표가 실제로 개선되는지 객관적으로 측정하기 위해서다.

회의에서는 많은 아이디어가 좋아 보일 수 있다.

- 버튼 색을 바꾸면 클릭률이 오를 것 같다.
- 추천 방식을 바꾸면 매출이 늘 것 같다.
- 결제 단계를 줄이면 전환율이 좋아질 것 같다.

하지만 중요한 것은 "정말 실제 사용자 행동이 바뀌는가"이다.  
A/B 테스트는 이 질문에 답하기 위한 실험이다.

즉, A/B 테스트는:

- 가설을 세우고
- 실제 사용자에게 두 버전을 비교 노출한 뒤
- 클릭률, 가입률, 구매율, 매출 같은 지표를 통해
- 변화의 효과를 검증하는 방법이다.

예:

- A: 현재 추천 방식
- B: 개인화 추천 방식

이때 비교할 수 있는 지표는 다음과 같다.

- 추천 클릭률
- 상품 상세페이지 진입률
- 구매 전환율
- 매출

이 과정을 통해 "좋아 보이는 아이디어"가 아니라  
"실제로 성과를 내는 변경"을 구분할 수 있다.

### 위험을 줄이기 위해서도 필요하다

A/B 테스트는 단순히 성과 측정 도구가 아니라, 위험을 줄이는 방법이기도 하다.

설문조사나 사용자 인터뷰에서 반응이 좋아도,  
실제 서비스 환경에서 사용자가 어떻게 행동할지는 다를 수 있다.

예:

- 설문에서는 "이 추천 기능 좋다"고 답했지만 실제로는 거의 클릭하지 않을 수 있다.
- 인터뷰에서는 "단계가 적은 결제가 편하다"고 말했지만 실제로는 정보 부족 때문에 불안해서 결제를 포기할 수도 있다.

그래서 새 기능을 전 사용자에게 한 번에 배포하는 대신,  
처음에는 작은 비율의 사용자에게만 노출해보는 것이 안전하다.

예:

- 5% 사용자에게만 새 기능 노출
- 문제 없으면 20%
- 성과가 괜찮으면 50%
- 이후 전체 배포

이렇게 하면 문제가 있을 때 전체 서비스에 미치는 영향을 줄일 수 있다.

## 13. 왜 애자일 A/B Test가 필요한가

여기서 말하는 "애자일"은 실험을 빠르게 반복할 수 있어야 한다는 뜻이다.  
즉, 실험 하나를 몇 달씩 끄는 것이 아니라 며칠 또는 몇 주 안에 설정, 실행, 분석, 의사결정이 돌아가야 한다는 의미다.

### 측정 방법도 중요하지만, 너무 오래 걸리면 문제다

A/B 테스트에서 지표를 어떻게 정의하고 측정할지는 매우 중요하다.  
하지만 그 과정이 너무 오래 걸리면 실험 자체의 가치가 떨어진다.

예를 들어:

- 실험 설정에 2주
- 로그 확인에 1주
- 분석 리포트 작성에 2주
- 배포 판단 회의에 1주

이렇게 되면 실험 하나에 1개월 이상이 걸릴 수 있다.

문제는 제품 개선은 보통 한 번의 실험으로 끝나지 않는다는 점이다.  
여러 번의 반복을 통해 조금씩 개선 방향을 찾아간다.

예:

- 1차 실험: 버튼 문구 변경
- 2차 실험: 버튼 위치 변경
- 3차 실험: 결제 단계 축소
- 4차 실험: 추천 모델 교체

만약 실험 한 번당 1개월이 걸리고, 의미 있는 개선까지 평균 6번의 반복이 필요하다면,  
하나의 기능을 제대로 출시하는 데 반년 가까이 걸릴 수도 있다.

### 애자일 A/B Test = 빠른 제품 개선

실험 시간이 줄어들수록 학습 속도가 빨라지고,  
학습 속도가 빨라질수록 제품 출시와 개선도 빨라진다.

즉:

- 빠른 실험
- 빠른 분석
- 빠른 의사결정
- 빠른 배포

가 가능한 팀일수록 제품을 더 빠르게 개선할 수 있다.

- Agile A/B Test
- Agile Product Improvement
- Agile Company

즉, 실험이 애자일해야 제품도 애자일해지고,  
제품이 애자일해야 회사 전체의 학습과 실행 속도도 빨라진다는 뜻이다.

### 특히 ML 기반 A/B Test에서는 더 중요하다

프론트엔드 문구나 버튼 색 실험은 비교적 가볍게 돌릴 수 있다.  
하지만 ML 기반 A/B 테스트는 훨씬 복잡하다.

예:

- 새로운 추천 모델 학습
- 실험군 사용자에게만 모델 노출
- 실험 로그 수집
- 클릭률/구매율 분석
- 성과가 좋으면 실제 배포

이 과정이 모두 수동이면 실험 속도가 너무 느려진다.  
그래서 ML 기반 실험에서는 아래 자동화가 특히 중요하다.

- 실험 설정 자동화
- 실험 로그 수집 자동화
- 분석 자동화
- 모델 배포 자동화

즉, ML 기반 A/B 테스트에서는 전체 실험 프로세스를 자동화하는 것이 거의 필수에 가깝다.

## 14. 짧게 정리

- A/B 테스트는 비즈니스 지표가 실제로 개선되는지 객관적으로 확인하기 위해 한다.
- 설문이나 직관만으로는 실제 사용자 행동을 정확히 알 수 없기 때문에 필요하다.
- 일부 사용자에게만 먼저 노출함으로써 위험을 줄일 수 있다.
- 하지만 실험이 너무 오래 걸리면 학습과 출시 속도가 느려진다.
- 그래서 실험 설정, 분석, 배포가 빠르게 돌아가는 애자일 A/B 테스트 체계가 중요하다.
- 특히 ML 기반 실험에서는 자동화가 매우 중요하다.
