# A/B 테스트 문제점과 주의사항

결과 분석과 시각화 관점은 아래 글에 이어서 정리했다.

- [A/B 테스트 결과 분석과 시각화]({% post_url 2026-03-28-ab-test-common-results-analysis-and-visualization %})

## 1. 왜 이 내용을 따로 정리해야 하는가

A/B 테스트는 개념만 이해한다고 잘 운영되는 것이 아니다.  
실제로는 실험 설계, 데이터 품질, 해석 방식, 동시 실험 간섭, 인프라 비용 같은 다양한 문제들이 생길 수 있다.

그래서 이 문서는 "A/B 테스트를 할 때 자주 생기는 문제"와  
"왜 그 문제가 위험한지", "어떻게 생각해야 하는지"를 따로 정리한 노트다.

## 2. 모든 결정을 데이터만으로 할 수 있는 것은 아니다

- 어떤 결정은 데이터로만 판단할 수 없다.
- 그래서 `Data Driven Decision`만이 아니라 `Data Informed Decision`이 필요하다.

여기서 의미는 이렇다.

- `Data Driven`: 데이터가 거의 전부를 결정하는 방식
- `Data Informed`: 데이터를 매우 중요하게 보되, 맥락과 전략과 사용자 경험도 함께 고려하는 방식

예를 들어:

- 브랜드 이미지 훼손 가능성이 있는 실험
- 장기 신뢰에 영향을 주는 가격 표시 방식
- 법적/윤리적 이슈가 있는 실험

이런 건 단기 클릭률이나 전환율만 보고 결정하면 안 된다.

예:

- 과장된 문구를 넣었더니 클릭률은 올랐다
- 하지만 사용자가 낚였다고 느껴 장기 신뢰가 떨어질 수 있다

이 경우 숫자만 보면 성공처럼 보일 수 있지만, 실제로는 좋은 의사결정이 아닐 수 있다.

즉, A/B 테스트는 매우 강력한 도구이지만  
모든 문제를 대신 판단해주는 만능 도구는 아니다.

## 3. 가설 없이 A/B Test를 남용하는 문제

즉, 모든 아이디어를 무조건 A/B 테스트로 돌리는 것은 좋은 접근이 아니다.

문제가 되는 경우:

- 가설이 없음
- 가설이 너무 대충 적혀 있음
- 왜 실험하는지 모호함
- 어떤 지표를 볼지 불명확함

예:

- "그냥 버튼 색 바꿔보자"
- "그냥 추천 UI 바꿔보자"

이런 식이면 결과가 좋아도 왜 좋아졌는지 배우기 어렵고,  
결과가 나빠도 무엇이 문제였는지 해석하기 어렵다.

좋은 실험은 최소한 아래가 있어야 한다.

- 무엇을 바꿀지
- 왜 바꾸는지
- 어떤 지표가 변할지
- 어느 방향으로 변할 것으로 기대하는지

## 4. 데이터 품질이 낮으면 실험도 믿기 어렵다

이 부분은 실제 운영에서 매우 자주 문제 된다.

- A/B 테스트 로그에 버그가 있으면 분석 품질이 무너진다
- Funnel 데이터에 버그가 있으면 결과 해석이 왜곡된다
- 표본 수가 충분한지도 확인해야 한다
- 버킷팅이 편향 없이 되었는지도 중요하다

### 4-1. 버킷 로그 오류

예:

- 실제로는 B를 본 사용자가 A로 기록됨
- 일부 사용자만 버킷 로그가 누락됨

이 경우 A/B 비교 자체가 틀어질 수 있다.

### 4-2. 행동 로그 오류

예:

- 클릭은 했는데 클릭 로그가 안 남음
- 구매는 했는데 매출 로그가 빠짐
- 모바일 앱만 impression 로그가 누락됨

이런 경우는 실험이 아니라 데이터 수집 체계가 문제인 것이다.

### 4-3. 표본 수 부족

실험 대상 사용자가 너무 적으면 우연한 변동이 커진다.

예:

- A 그룹 구매 2건
- B 그룹 구매 4건

겉으로는 두 배 차이처럼 보여도, 표본이 너무 적어서 의미 있는 차이라고 보기 어려울 수 있다.

즉, `sample size big enough?`는 반드시 확인해야 한다.

### 4-4. 편향된 버킷팅

버킷팅은 랜덤하고 균형 있게 이루어져야 한다.

예:

- 신규 사용자만 B에 더 많이 들어감
- iOS 사용자만 B에 몰림
- 특정 시간대 유입만 B에 더 많이 들어감

이러면 A/B 차이가 기능 차이 때문인지, 사용자 집단 차이 때문인지 구분이 안 된다.

즉, 실험군과 대조군은 가능한 한 공정하게 나뉘어야 한다.

## 5. 결과를 내 입맛대로 해석하는 문제

사람은 자기가 기대한 방향으로 결과를 해석하려는 경향이 있다.

예:

- 내가 만든 기능이라서 좋게 해석하고 싶다
- PM이 강하게 밀던 아이디어라 실패를 인정하기 어렵다
- 일부 지표만 좋으면 그것만 강조하고 싶어진다

이런 식이면 객관적 분석이 어렵다.

### 왜 최소 일주일 정도는 보라고 하는가

최소 일주일 정도는 보는 편이 좋은 이유는,  
요일 효과나 사용자 행동 패턴 차이가 있기 때문이다.

예:

- 평일과 주말 구매 패턴이 다를 수 있음
- 월초와 월말 소비 패턴이 다를 수 있음
- 특정 프로모션 날짜에만 트래픽이 급증할 수 있음

그래서 실험을 너무 빨리 끊으면 우연한 패턴을 결과로 착각할 수 있다.

### 왜 그룹 리뷰가 좋은가

결과 해석은 혼자 하지 않는 것이 좋다.

그룹 리뷰의 장점:

- 지표를 더 객관적으로 볼 수 있음
- 다른 팀이 놓친 문제를 발견할 수 있음
- 지식 공유가 됨
- 다음 실험의 학습으로 이어짐

즉, 실험 결과는 개인의 주장보다 팀의 학습 자산이 되어야 한다.

## 6. 여러 A/B Test 간 상호작용 문제

여러 실험을 동시에 돌리면 각 실험이 서로 영향을 줄 수 있다.

예:

- 장바구니 페이지에서 A/B 테스트 진행
- 체크아웃 페이지에서도 동시에 A/B 테스트 진행

이때 어떤 사용자는:

- 장바구니에서는 B
- 체크아웃에서는 A

를 볼 수 있고, 또 다른 사용자는:

- 장바구니에서는 B
- 체크아웃에서는 B

를 볼 수 있다.

그럼 결과가 섞인다.

예:

- 체크아웃 개선 때문인지
- 장바구니 개선 때문인지
- 둘이 동시에 적용돼서 효과가 난 건지

구분하기 어려워진다.

이런 상호작용은 발견하기도 쉽지 않다.

### 가장 안전한 방법

가장 안전한 방식은 한 번에 하나의 실험만 돌리는 것이다.

하지만 현실에서는 너무 느릴 수 있다.  
그래서 실무에서는:

- 서로 완전히 다른 영역인지
- 같은 KPI에 영향을 주는지
- 같은 사용자 흐름 안에 있는지

를 보고 동시에 돌릴지 판단한다.

상호작용이 강한 경우에는 단순 A/B가 아니라  
`Multivariate Test` 관점이 필요할 수도 있다.

## 7. 데이터 인프라 비용 문제

A/B 테스트는 생각보다 운영 비용이 크다.

A/B 테스트 분석 파이프라인은  
매우 큰 데이터 인프라 비용을 차지할 수 있다.

왜냐하면 실험을 잘 하려면 아래가 필요하기 때문이다.

- 버킷 로그 저장
- 사용자 행동 로그 저장
- 실험별 조인 파이프라인
- 대시보드 계산
- 세그먼트별 분석
- 장기 지표 추적

즉, A/B 테스트는 단순 기능 실험이 아니라 데이터 시스템을 계속 쓰는 작업이다.

실험이 많아질수록:

- 저장 비용
- 쿼리 비용
- ETL/ELT 비용
- 대시보드 유지 비용

도 같이 커진다.

그래서 모든 아이디어를 무조건 실험으로 돌리기보다  
중요한 실험에 집중하는 것이 필요하다.

## 8. 비교 대상이 공정하지 않은 문제

비교 자체가 공정하지 않으면 실험 결론도 믿기 어렵다.

대표적인 경우는 두 가지다.

### 8-1. 한 번에 여러 변화를 같이 넣는 경우

예:

- UI 디자인도 바꿈
- 추천 알고리즘도 같이 바꿈

이렇게 하면 결과가 좋아졌을 때:

- UI 때문인지
- 추천 모델 때문인지
- 둘의 조합 때문인지

구분이 안 된다.

즉, 가능하면 한 실험에서는 핵심 변화 하나에 집중하는 것이 좋다.

### 8-2. Control users와 Test users가 사실상 다를 때

예:

- A에는 주로 기존 충성 사용자가 많고
- B에는 신규 사용자가 더 많음

이 경우 기능 차이보다 사용자 집단 차이가 결과를 만든 것일 수 있다.

그래서 버킷 배정의 공정성과 사용자 속성 균형이 매우 중요하다.

## 9. 실험을 얼마나 오래 볼 것인가 문제

- "언제 이 실험이 성공인지 실패인지 결정할 것인가?"

너무 빨리 결정하면 우연한 잡음을 진짜 효과로 착각할 수 있다.  
너무 오래 기다리면 제품 개선 속도가 느려진다.

즉, 실험 기간은 짧아도 문제고 길어도 문제다.

보통 아래를 같이 고려한다.

- 최소 필요한 표본 수
- 평일/주말 패턴
- 결제/재방문 같은 지표의 관측 지연
- 비즈니스 의사결정 속도

예:

- 클릭률 실험은 며칠 내 신호가 보일 수 있음
- 구매율이나 리텐션은 더 오래 봐야 할 수 있음

즉, "얼마나 기다릴지"도 실험 설계의 일부다.

## 10. 실무적으로 체크해야 할 질문

A/B 테스트를 시작하거나 해석하기 전에 아래 질문을 체크하면 좋다.

- 이 실험은 정말 데이터로 판단 가능한 문제인가?
- 가설이 충분히 구체적인가?
- 버킷 로그와 행동 로그 품질은 믿을 만한가?
- 표본 수는 충분한가?
- 버킷팅은 편향 없이 되었는가?
- 동시에 도는 다른 실험과 상호작용은 없는가?
- 한 번에 너무 많은 변경을 넣고 있지 않은가?
- 결과를 충분한 기간 동안 관찰했는가?
- 결과를 팀 단위로 객관적으로 리뷰했는가?

## 11. 한 줄 정리

A/B 테스트의 가장 큰 위험은 "실험을 했으니 결과도 믿을 수 있다"고 착각하는 것이다.  
실제로는 가설, 데이터 품질, 버킷 공정성, 실험 간 간섭, 해석 태도, 관찰 기간까지 모두 잘 관리되어야 실험 결과를 믿을 수 있다.
