# A/B 테스트 질문 리뷰

보충 복습과 딥다이브 내용은 아래 글에 이어서 정리했다.

- [A/B 테스트 리뷰 딥다이브]({% post_url 2026-03-28-ab-test-common-review-deep-dive %})

## 1. 왜 이 문서를 따로 정리하는가

A/B 테스트를 공부하다 보면 개념 자체보다  
실무적으로 "그래서 실제로는 어떻게 하는가?"가 더 궁금해질 때가 많다.

이 문서는 그런 질문들에 대해  
지금까지 정리한 개념을 바탕으로 답을 붙인 학습용 리뷰 노트다.

질문에 따라:

- 비교적 명확하게 답할 수 있는 것
- 회사나 상황에 따라 달라지는 것
- 정확한 내부 정보가 없어서 일반론으로만 답할 수 있는 것

을 구분해서 정리했다.

## 2. A/B 테스트 비용의 70%는 무엇을 뜻하는가

질문:

- A/B 테스트에서 70%의 비용이 사용된다고 했을 때, 이 비용은 SaaS 비용만 뜻하는가?
- 인력 비용도 포함되는가?
- 직접 구현하면 이 비용이 몇 %로 줄어드는가?

답:

이런 종류의 비용은 보통 SaaS 구독료만 뜻하지 않는다.  
실무에서는 보통 아래가 모두 비용이다.

- 실험 플랫폼 구축 비용
- 로그 저장 비용
- ETL/ELT 비용
- 분석용 쿼리 비용
- 대시보드 운영 비용
- 엔지니어링 인력 비용
- 데이터 분석가/데이터 엔지니어 인력 비용
- QA와 운영 비용

즉, A/B 테스트 비용은 `도구 비용 + 데이터 인프라 비용 + 사람의 시간`이 합쳐진 개념으로 보는 편이 자연스럽다.

### 직접 구현하면 비용이 줄어드는가

단기적으로는 오히려 늘 수 있다.

이유:

- 처음에는 직접 플랫폼을 만드는 개발 비용이 크고
- 운영 체계를 갖추는 데 시간이 들고
- 품질을 안정화하는 데도 비용이 든다

장기적으로는 회사 상황에 따라 줄어들 수 있다.

예:

- 실험 수가 많아 SaaS 요금이 비싸질 때
- 내부 데이터와 SaaS 연결 비용이 반복적으로 들 때
- 회사가 원하는 방식으로 커스터마이징이 많이 필요할 때

즉, 직접 구현이 무조건 더 싸다고 보기 어렵고  
`초기에는 비싸고, 규모가 커지면 유리해질 수 있다` 정도로 이해하는 것이 좋다.

### 몇 %로 줄어드는가

이건 일반화하기 어렵다.

이유:

- 트래픽 규모
- 실험 수
- 조직의 인건비 구조
- 기존 데이터 인프라 수준
- SaaS 계약 단가

가 모두 다르기 때문이다.

즉, 특정 퍼센트로 외워두기보다는  
`A/B 테스트 비용의 대부분은 단순 툴 사용료가 아니라 운영 전체 비용`이라고 이해하는 것이 더 중요하다.

## 3. `LEFT`나 나머지 연산에서 버려지는 부분은 문제가 없나

질문:

- A/B를 나눌 때 `LEFT`를 하거나 나머지 연산을 할 때 버려지는 부분이 중복되거나 왜곡되지 않는가?
- 50:50에 영향이 적어서 무시하는 것인가?

답:

핵심은 `버려지는 부분`이 중요한 것이 아니라  
최종적으로 얻는 버킷 번호가 균등하고 안정적인가이다.

예를 들어:

```python
bucket = hash(user_id) % 2
```

이 코드는 해시값 전체를 다 쓰지 않고  
마지막에 `% 2` 결과만 사용한다.

여기서 중요한 점은:

- 같은 `user_id`는 항상 같은 해시값을 만들고
- 같은 해시값은 항상 같은 버킷 번호를 만들며
- 많은 사용자에 대해 보면 결과가 비교적 고르게 분산된다는 점이다

즉, 앞부분이 버려진다고 해서 문제가 되는 것이 아니라  
해시값이 충분히 잘 섞여 있다면 `% 2`나 `% 100` 같은 연산으로도 안정적으로 버킷팅할 수 있다.

### 중복이 문제 아닌가

중복 자체는 문제 아니다.

왜냐하면 원래 목적이 `모든 사용자를 서로 다른 숫자로 만드는 것`이 아니라  
`사용자를 정해진 수의 버킷에 나누는 것`이기 때문이다.

A/B 테스트에서는 같은 버킷에 여러 사용자가 들어가는 것이 정상이다.

중요한 것은:

- 한쪽으로 과도하게 몰리지 않는가
- 같은 사용자가 매번 같은 버킷에 들어가는가

이다.

## 4. 데이터 문해력을 높이기 위한 팀 문화는 무엇이 있나

질문:

- 팀에서 데이터 문해력을 높이기 위해 어떤 프로세스나 문화를 운영하면 좋은가?

답:

실무적으로 효과적인 방법은 보통 아래와 같다.

- 주간 A/B 테스트 리뷰 미팅 운영
- 실험 결과를 공개적으로 공유
- 지표 정의 문서화
- 대시보드 정의 통일
- 가설 작성 템플릿 사용
- 실험 실패 사례도 공유
- 분석 리뷰를 특정 개인에게만 의존하지 않기

### 왜 이런 활동이 중요한가

데이터 문해력은 단순히 SQL을 잘하는 능력이 아니다.

- 어떤 지표를 믿어야 하는지
- 어떤 해석이 위험한지
- 어떤 실험이 좋은 실험인지

를 팀이 같이 배우는 능력에 가깝다.

그래서 데이터 문해력을 높이려면  
`도구 교육`보다 `반복적인 리뷰 문화`가 더 중요할 때가 많다.

## 5. 실험은 1%에서 유의미하면 바로 끝내도 되는가

질문:

- 1%나 5%에서 유의미한 결과가 나오면 바로 테스트를 멈추고 의사결정하는가?
- 아니면 5% -> 30% -> 100%처럼 점차 늘리면서 확인하는가?

답:

보통 1%나 5%는 `효과를 최종 확정하는 단계`라기보다  
`시스템 안정성과 초기 신호를 보는 단계`에 가깝다.

즉, 이 단계에서 주로 보는 것은:

- 기능이 정상 동작하는가
- 로그가 잘 쌓이는가
- 치명적인 매출 하락이나 오류가 없는가
- 아주 이상한 결과가 없는가

이다.

### 유의미하면 바로 끝내는가

상황에 따라 다르지만, 보통은 바로 끝내기보다  
더 큰 트래픽에서 다시 확인하는 경우가 많다.

이유:

- 표본이 아직 적을 수 있고
- 초기 사용자 집단이 전체를 대표하지 않을 수 있고
- 롤아웃 초기에만 나타나는 우연한 패턴일 수 있기 때문이다

즉, 보통은:

1. 작은 비율에서 안정성 확인
2. 점차 비율 확대
3. 충분한 표본과 기간이 확보되면 최종 판단

순서로 가는 것이 자연스럽다.

## 6. 3%에서는 유의했는데 5%에서는 약해지면 어떻게 하나

질문:

- 초반 소규모 트래픽에서는 유의미해 보였는데, 더 늘리니 차이가 줄어들면 어떻게 해석해야 하나?

답:

이 경우 초반 결과가 우연이었을 가능성을 먼저 생각해야 한다.

소규모 표본에서는:

- 우연한 변동이 커지고
- 특정 사용자 몇 명의 영향이 커지며
- 일부 세그먼트 효과가 전체 효과처럼 보일 수 있다

그래서 트래픽을 늘렸더니 효과가 줄어드는 것은 이상한 일이 아니다.

### 보통 어떻게 하나

- 실험 기간을 더 가져가고
- 표본을 더 확보하고
- 일별 추세와 세그먼트별 결과를 같이 보고
- outlier나 bot 영향이 없는지 점검한다

즉, 작은 표본에서 유의했다고 바로 결론 내리기보다  
더 큰 데이터에서 재확인하는 것이 중요하다.

## 7. A와 B의 사용자 수가 다르면 샘플링으로 맞춰도 되는가

질문:

- A 집단이 더 크면 샘플링으로 줄여서 B와 비율을 맞춰도 되는가?

답:

보통은 원래 배정된 데이터를 그대로 보는 것이 우선이다.

이유:

- A/B 테스트는 처음 버킷 배정 자체가 랜덤하게 이루어졌다는 전제를 활용하는데
- 사후에 샘플링을 다시 하면 추가 편향이 들어갈 수 있기 때문이다

즉, 이미 랜덤하게 나뉜 집단을  
분석 단계에서 다시 손대는 것은 조심해야 한다.

### 사용자 수 차이가 조금 나면 문제인가

조금 차이 나는 것은 자연스러울 수 있다.

예:

- 49.7% vs 50.3%

이 정도는 보통 큰 문제가 아니다.

중요한 것은 단순한 숫자 차이보다:

- 사용자 속성 균형이 비슷한가
- 특정 세그먼트가 한쪽에 몰리지 않았는가

이다.

### 언제 문제가 되나

- 버킷 비율 차이가 너무 크거나
- 특정 버전에서 로그 누락이 있거나
- 버킷팅 로직 자체가 깨졌을 때

이런 경우는 샘플링보다 먼저  
배정 로직과 데이터 품질을 의심해야 한다.

## 8. A/B 테스트는 대고객 조직만 하는 것인가

질문:

- 대고객 서비스를 맡지 않은 조직이나 데이터 엔지니어 조직은 A/B 테스트를 할 일이 없는가?

답:

반드시 그렇지는 않다.

A/B 테스트는 꼭 외부 고객 대상 서비스에만 쓰는 기법이 아니다.

예:

- 사내 검색 도구 개선
- 내부 대시보드 UX 개선
- 업무 자동화 도구 사용성 비교
- 알림 방식 변경에 따른 응답 시간 비교

이런 것도 실험 대상이 될 수 있다.

다만 조건이 있다.

- 비교할 버전이 있어야 하고
- 측정할 지표가 있어야 하며
- 실험할 만큼 사용량이 있어야 한다

즉, 사용자만 있다면 그 사용자가 외부 고객이든 내부 직원이든  
실험 자체는 가능하다.

## 9. 초창기 스타트업은 A/B 테스트 도입이 어려운가

질문:

- 가설과 데이터가 필요하니 초창기 스타트업에서는 A/B 테스트가 어렵다고 봐야 하는가?

답:

대체로 맞는 이해다.

초창기 스타트업은 보통:

- 트래픽이 적고
- 전환 수가 적고
- 제품 구조가 빠르게 바뀌며
- 실험 인프라를 만들 여력이 부족하다

그래서 정교한 A/B 테스트를 운영하기 어려울 수 있다.

### 그렇다면 아무것도 못 하나

그렇지는 않다.

초기에는 보통 아래 방법이 더 유용하다.

- 사용자 인터뷰
- 세일즈 콜
- 프로토타입 테스트
- 설문조사
- 정성적 피드백 수집

즉, 초기에는 `가설 검증`은 필요하지만  
꼭 정식 A/B 테스트일 필요는 없다.

트래픽과 제품이 어느 정도 안정되면  
그때 A/B 테스트를 본격 도입하는 것이 자연스럽다.

## 10. 어떤 아이디어가 A/B 테스트할 만한가

질문:

- A/B 테스트 미팅에서 어떤 아이디어를 채택하고 어떤 아이디어를 기각하는가?

답:

보통 아래 질문에 많이 걸린다.

- 가설이 구체적인가
- 비교 가능한 두 버전이 있는가
- 성공 지표가 명확한가
- 실험 트래픽이 충분한가
- 구현 비용 대비 배울 것이 큰가
- 실패해도 감당 가능한 리스크인가
- 다른 더 쉬운 검증 방법이 먼저 필요한가

### 기각되기 쉬운 아이디어

- 그냥 재미있어 보여서 해보는 것
- 무엇을 배울지 모호한 것
- 동시에 너무 많은 변화가 들어가는 것
- 지표가 명확하지 않은 것
- 트래픽이 너무 적은 것

즉, 좋은 실험 아이디어는  
`구체적 가설 + 명확한 지표 + 실행 가능성 + 학습 가치`를 함께 가져야 한다.

## 11. 머신러닝 추천으로 바꾸면 마케터는 어떻게 되는가

질문:

- 마케터 중심 규칙 기반 추천을 머신러닝 추천으로 바꾸면 마케터는 어떻게 되는가?

답:

이건 회사마다 다르고, 외부에서 정확히 알기 어려운 내부 운영 문제다.

그래서 특정 회사에서 실제로 어떻게 되었는지는  
추측보다 확인 가능한 정보가 있을 때만 단정하는 것이 맞다.

다만 일반적으로는 역할이 사라진다기보다 바뀌는 경우가 많다.

예:

- 추천 규칙을 직접 일일이 만드는 역할은 줄어들고
- 프로모션 전략 설계
- 추천 영역 기획
- 세그먼트 정의
- 콘텐츠 큐레이션
- 캠페인 운영

같은 역할이 더 중요해질 수 있다.

즉, 머신러닝이 들어오면 사람의 역할이 완전히 없어지기보다  
`수동 규칙 작성`에서 `전략, 운영, 해석` 쪽으로 이동하는 경우가 많다.

## 12. 짧게 다시 정리

- A/B 테스트 비용은 SaaS 비용만이 아니라 인력과 데이터 인프라 비용까지 포함한다.
- 직접 구현은 단기적으로 더 비쌀 수 있고, 장기적으로 유리해질 수 있다.
- 해시와 나머지 연산에서 중요한 것은 버려지는 숫자가 아니라 균등하고 안정적인 분배다.
- 작은 트래픽에서 유의미해 보여도 더 큰 표본에서 다시 확인해야 한다.
- A/B 집단 크기가 조금 다른 것은 자연스러우며, 사후 샘플링은 조심해야 한다.
- A/B 테스트는 내부 서비스에도 적용할 수 있다.
- 초창기 스타트업은 정교한 A/B 테스트보다 인터뷰와 정성 검증이 더 적합할 수 있다.
- 좋은 실험 아이디어는 구체적인 가설과 명확한 지표를 가져야 한다.
