# 가상 A/B 테스트 데이터 분석과 Two-Sample t-test

## 1. 왜 이 내용을 따로 정리해야 하는가

앞에서는:

- 트래픽이 제대로 나뉘었는지
- 버킷 크기가 정상인지
- proportion z-test로 50:50 분배가 맞는지

를 먼저 점검했다.

이제부터는 한 단계 더 나아가  
실제 지표를 A와 B 사이에서 비교해야 한다.

즉:

- Impression
- Click
- Purchase
- Paidamount

같은 지표를 두 집단 간에 비교하는 단계다.

이 문서는 가상 A/B 테스트 데이터를 바탕으로  
Two-Sample t-test를 어떻게 이해하고 사용하는지 정리한 노트다.

## 2. 이 파트의 핵심 질문은 무엇인가

트래픽 분배가 정상이라고 확인되었다면  
이제 질문은 바뀐다.

- A와 B의 평균 impression이 다른가
- A와 B의 평균 click이 다른가
- A와 B의 평균 purchase가 다른가
- A와 B의 평균 paidamount가 다른가

즉, 이제는 `비율 자체`가 아니라  
`두 집단의 평균 지표 차이`를 보는 문제다.

## 3. 이때 왜 Two-Sample T-Test를 쓰나

이 문제는 두 개의 독립적인 집단을 비교하는 문제다.

- 집단 1: control
- 집단 2: test

그리고 각 집단 안에는:

- 사용자별 값
- 세션별 값
- 날짜별 요약값

같은 관측치가 여러 개 있다.

즉, `두 샘플의 평균 차이`를 보는 문제이므로  
Two-Sample t-test가 자연스럽게 등장한다.

## 4. Two-Sample T-Test는 무엇을 검정하나

귀무가설은 보통 아래처럼 놓는다.

- `H0: mean_A = mean_B`

즉:

- A의 평균 클릭 수와 B의 평균 클릭 수는 같다
- A의 평균 구매 수와 B의 평균 구매 수는 같다
- A의 평균 매출액과 B의 평균 매출액은 같다

라고 두고 시작한다.

그 다음:

- 실제로 관측된 평균 차이가
- 우연한 흔들림으로도 설명 가능한지

를 검정한다.

좀 더 수식으로 쓰면 보통 이렇게 본다.

- `H0: μA = μB`
- 또는 `H0: μB - μA = 0`

즉, 두 집단의 진짜 평균 차이가 0이라고 놓고  
지금 샘플에서 본 차이가 그 가정과 잘 맞는지 확인하는 것이다.

여기서 중요한 점은  
우리가 직접 아는 것은 `진짜 평균`이 아니라  
샘플에서 계산한 평균 `mean_a`, `mean_b`라는 점이다.

그래서 결국 t-test는  
`샘플 평균 차이`를 보고  
`진짜 평균 차이도 있다고 볼 수 있는가`를 추론하는 과정이다.

## 5. t-test의 결과는 결국 무엇인가

Two-Sample t-test의 결과도 결국  
`차이가 얼마나 극단적인지`를 나타내는 점수와  
그 차이가 우연히 나올 확률을 준다.

즉:

- t-score
- p-value

가 핵심 결과다.

실무에서는 t-score를 보기도 하고,  
p-value를 기준으로 유의성을 판단하기도 한다.

공식 형태는 아래처럼 볼 수 있다.

```text
t = (mean_b - mean_a) / SE(mean_b - mean_a)
```

즉:

- 분자 `mean_b - mean_a`는 실제로 관측한 평균 차이
- 분모 `SE(mean_b - mean_a)`는 그 차이가 원래 얼마나 흔들릴 수 있는지

를 뜻한다.

결국 t-score도 z-score와 마찬가지로  
`관측한 차이 / 원래 흔들림 크기`  
형태라는 점이 핵심이다.

## 5-1. 그럼 왜 이름은 t-test인가

여기서 많은 사람이 헷갈린다.

앞에서 본 z-test는 보통 표준정규분포를 기준으로 읽는다.  
반면 t-test는 `t-distribution`을 기준으로 읽는다.

왜냐하면 실무에서는 보통 모집단 표준편차를 정확히 모르고,  
샘플 데이터로 분산을 추정해야 하기 때문이다.

즉:

- 정규분포와 전혀 다른 개념이 아니라
- 평균 차이를 표준화해서 본다는 큰 구조는 같고
- 분산을 샘플로 추정하는 불확실성을 더 반영한 검정이 t-test다

그래서 t-distribution은  
표준정규분포보다 꼬리가 조금 더 두껍다.

샘플 수가 커질수록 t-distribution은 표준정규분포에 가까워진다.

## 6. 왜 95% 신뢰도를 주로 쓰나

온라인 실무에서는 보통 95% 신뢰도를 많이 쓴다.

즉:

- 유의수준 5%
- p-value < 0.05

를 자주 기준으로 삼는다.

왜 99%를 잘 안 쓰냐면  
기준을 너무 엄격하게 높이기보다  
샘플 크기를 충분히 확보하는 것이 보통 더 실용적이기 때문이다.

즉:

- 95%는 실무에서 많이 쓰는 기본 기준
- 더 엄격한 기준보다 충분한 샘플 확보가 더 중요

라고 이해하면 된다.

## 7. z-score가 1.95면 어떻게 해야 하나

이 질문은 실무에서 자주 나온다.

95% 양측검정 기준에서  
경계값은 보통 `1.96`이다.

즉:

- `1.95`는 아주 근접했지만 기준을 넘지 못한 값이다

이 경우를 임의적으로  
"거의 유의하다"라고 해석하는 것은 조심해야 한다.

이럴수록 중요한 것은:

- 더 많은 샘플 확보
- 실험 기간 연장
- 미리 정한 종료 기준 유지

다.

즉, 경계값 근처라고 해서  
사후적으로 해석을 바꾸지 않는 것이 중요하다.

## 8. Data Peeking은 왜 위험한가

실험을 돌리면서 매일 결과를 보다가  
유의해 보이는 순간 멈추는 것을 `data peeking`이라고 한다.

이건 위험하다.

왜냐하면:

- 초반에는 우연한 차이가 크게 보일 수 있고
- 시간이 지나면 차이가 줄어들 수도 있기 때문이다

즉, 실험은 미리 정한 기준과 충분한 샘플을 확보한 뒤에 해석해야 한다.

## 9. 이상 데이터는 왜 따로 봐야 하나

A/B 테스트에서는 이상 데이터가 결과를 크게 왜곡할 수 있다.

대표 예:

- bot traffic
- whale user
- 비정상적 click/impression 비율

이런 데이터는 특정 실험에만 임의로 적용하기보다  
모든 A/B 테스트에 일관된 규칙으로 처리하는 것이 좋다.

즉, outlier 처리도 실험 해석의 일부가 아니라  
분석 시스템의 표준 운영 규칙으로 가는 것이 더 좋다.

## 10. 가상 A/B 테스트 데이터는 어떤 구조를 가지나

이 파트에서 사용하는 가상 데이터는 보통 세 가지 테이블로 구성된다.

- `raw_data.user_event`
- `raw_data.user_variant`
- `raw_data.user_metadata`

## 11. `raw_data.user_event`는 무엇인가

이 테이블은 사용자/날짜/아이템 수준에서  
행동 이벤트를 기록한 테이블이다.

예 컬럼:

- `user_id`
- `datestamp`
- `item_id`
- `clicked`
- `purchased`
- `paidamount`

즉, impression이 발생한 단위에서:

- 클릭했는지
- 구매했는지
- 얼마를 결제했는지

를 기록한 원천 데이터다.

실무에서는 이런 데이터를 로그 파일이나 이벤트 스트림에서 적재한 뒤  
분석용 테이블로 다시 요약하는 경우가 많다.

## 12. `raw_data.user_variant`는 무엇인가

이 테이블은 사용자가 어떤 variant에 속하는지 기록한다.

예:

- `user_id`
- `variant_id`

보통은:

- control
- test

같은 구분이 들어간다.

실무에서는 experiment 테이블과 variant 테이블이 분리되어 있거나,  
variant 소속 시점 타임스탬프가 함께 기록되는 경우도 많다.

즉, 이 테이블은  
실험 배정 정보의 출처라고 보면 된다.

## 13. `raw_data.user_metadata`는 무엇인가

이 테이블은 사용자 메타 정보를 담는다.

예:

- `user_id`
- `age`
- `gender`

이런 정보는:

- 연령대별 비교
- 성별 비교
- 세그먼트별 실험 결과 분석

에 활용된다.

즉, 전체 평균만 보지 않고  
사용자 그룹별로 다시 나눠볼 수 있게 해주는 데이터다.

## 14. 왜 요약 테이블을 따로 만드나

원천 이벤트 테이블을 그대로 두고 바로 t-test를 하기는 어렵다.

그래서 보통 아래처럼 요약 테이블을 만든다.

예:

```sql
CREATE TABLE analytics.variant_daily_sessions AS
SELECT
    variant_id,
    user_id,
    datestamp,
    count(distinct item_id) num_of_items,
    sum(clicked) num_of_clicks,
    sum(purchased) num_of_purchases,
    sum(paidamount) revenue
FROM raw_data.user_event ue
JOIN raw_data.user_variant uv ON ue.user_id = uv.user_id
GROUP BY 1, 2, 3;
```

## 15. 이 요약 테이블은 왜 중요한가

이 테이블은:

- variant별
- 사용자별
- 날짜별

통계 정보를 만들어준다.

즉, 이후 분석에서:

- impression 평균
- click 평균
- purchase 평균
- revenue 평균

을 비교하기 쉬운 형태로 데이터를 바꿔놓는 것이다.

실무에서는 이런 요약 테이블을  
ELT 파이프라인으로 만드는 경우가 많다.

## 16. Two-Sample T-Test에서 무엇이 필요한가

두 집단 평균을 비교하려면 보통 아래 값이 필요하다.

- `mean_a`, `mean_b`
- `n_a`, `n_b`
- `var_a`, `var_b`

즉:

- 평균
- 샘플 수
- 분산

이 세 가지가 핵심이다.

조금 더 정확히 말하면  
평균 차이의 표준오차를 계산할 수 있어야 한다.

그래서 결국 필요한 값은:

- 각 집단 평균
- 각 집단 표본 수
- 각 집단 분산

이고, 이 값들로  
`두 평균 차이가 원래 얼마나 흔들릴 수 있는지`를 계산한다.

즉, two-sample t-test는  
단순히 평균 두 개를 빼보는 것이 아니라  
그 차이를 표준오차로 표준화하는 과정이다.

## 17. `paidamount`를 예로 들면 어떻게 되나

A 집단과 B 집단 각각에 대해 아래를 계산할 수 있다.

### Bucket A

- `n_a`: 세션 수
- `sum_a`: 매출액 합
- `mean_a`: 평균 매출액
- `sum2_a`: 매출액 제곱합
- `square_a`: 매출액 제곱평균
- `var_a`: 분산

### Bucket B

- `n_b`: 세션 수
- `sum_b`: 매출액 합
- `mean_b`: 평균 매출액
- `sum2_b`: 매출액 제곱합
- `square_b`: 매출액 제곱평균
- `var_b`: 분산

즉, 각 집단의 평균과 분산을 따로 계산한 뒤  
그 차이를 검정하는 구조다.

## 18. t-score는 어떻게 계산하나

핵심은 다음 식이다.

```python
t_score = (mean_b - mean_a) / math.sqrt(var_a / n_a + var_b / n_b)
```

즉:

- 분자: 평균 차이
- 분모: 그 차이가 흔들릴 수 있는 정도

가 된다.

이 값이 크면  
평균 차이가 우연한 흔들림보다 크다는 뜻이다.

수식 형태로 쓰면 아래와 같다.

```text
t = (mean_b - mean_a) / sqrt(var_a / n_a + var_b / n_b)
```

이 식은 실무에서 자주 쓰는 Welch 형태에 가깝다.

즉:

- A와 B의 분산이 꼭 같다고 강하게 가정하지 않고
- 각 집단 분산을 따로 반영해서 비교하는 방식이다

그래서 실무 A/B 테스트에서는  
이 형태가 더 자연스럽게 받아들여지는 경우가 많다.

이 공식을 보면 해석도 바로 따라온다.

- 평균 차이가 크면 분자가 커져 t-score가 커진다
- 샘플 수가 많아지면 분모가 작아져 t-score가 커진다
- 분산이 크면 분모가 커져 t-score가 작아진다

즉, t-score는  
`평균 차이가 원래 데이터 흔들림에 비해 얼마나 큰가`를 숫자로 만든 것이다.

## 19. 왜 결국 이 값도 z-score처럼 해석하나

실무에서 Two-Sample t-test를 설명할 때  
t-score를 사실상 z-score처럼 직관적으로 해석하는 경우가 많다.

즉:

- 0 근처면 차이가 약함
- 절댓값이 커질수록 차이가 큼

이라고 본다.

95% 양측 기준에서는 보통:

- `-1.96 ~ 1.96`

안쪽이면 큰 차이가 아니라고 보고,  
그 밖이면 유의미한 차이라고 보는 식으로 설명한다.

이런 설명이 가능한 이유는  
샘플 수가 충분히 크면 t-distribution이 표준정규분포에 가까워지기 때문이다.

즉 입문 단계에서는  
`t-score도 결국 기준 분포 위에서 얼마나 극단적인지를 보는 값`  
으로 이해하면 흐름이 잘 이어진다.

다만 더 엄밀하게는:

- t-test는 t-distribution을 기준으로 p-value를 계산하고
- 자유도(degrees of freedom)에 따라 임계값도 조금 달라질 수 있다

즉, 실무 설명에서는 z-score처럼 직관적으로 받아들이되,  
통계적으로는 완전히 같은 것은 아니라는 점도 같이 기억하면 좋다.

## 19-1. 그럼 t-test는 언제 특히 잘 맞나

Two-Sample t-test는 아래 같은 상황에 특히 잘 맞는다.

- 두 독립 집단이 있다
- 각 집단 평균을 비교하고 싶다
- 모집단 분산은 알지 못한다
- 샘플 수가 어느 정도 있다

A/B 테스트에서는 보통:

- control vs test
- 사용자별 또는 세션별 요약값
- 클릭 수, 구매 수, 매출액 같은 연속형 또는 준연속형 지표

비교에 자주 쓰인다.

## 20. Python에서는 어떻게 계산하나

Python에서는 보통 `scipy.stats.ttest_ind`를 많이 사용한다.

예:

```python
tscore, pvalue = stats.ttest_ind(b, a)
print(tscore, pvalue)
```

즉, 각 집단의 raw value 리스트를 넣으면  
t-score와 p-value를 바로 계산할 수 있다.

실무에서는 보통 Welch t-test를 많이 쓰므로  
아래처럼 `equal_var=False`를 주는 것도 자주 본다.

```python
tscore, pvalue = stats.ttest_ind(b, a, equal_var=False)
```

이 옵션은:

- A와 B의 분산이 꼭 같다고 가정하지 않는 방식
- 현실 데이터에서 더 안전한 기본값으로 많이 사용되는 방식

으로 이해하면 된다.

실무에서는:

- 직접 공식으로 계산해보는 방법
- 라이브러리로 계산하는 방법

두 가지를 모두 이해하는 것이 좋다.

## 21. 왜 직접 계산도 해봐야 하나

라이브러리를 쓰면 편하지만,  
직접 계산을 이해해야 Tableau나 SQL, PySpark처럼  
다른 환경에서도 똑같은 논리를 구현할 수 있다.

즉, 직접 계산을 이해하면:

- BI 툴에서 동적으로 계산할 때
- SQL로 옮길 때
- Spark로 대량 데이터 처리할 때

더 유연하게 대응할 수 있다.

## 22. 어떤 지표에 반복 적용하나

Two-Sample t-test는 아래 지표에 반복해서 적용할 수 있다.

- Impression
- Click
- Purchase
- Paidamount

즉, 지표마다:

- A 평균
- B 평균
- t-score
- p-value

를 계산해 비교할 수 있다.

## 23. 결과는 어떻게 보여주나

실무에서는 보통:

- A 평균
- B 평균
- B-A 차이
- t-score 또는 p-value

를 같이 보여준다.

그리고 유의성이 있는 경우  
컬러 코딩으로 강조하는 경우가 많다.

즉, 분석은 숫자 계산만이 아니라  
해석이 쉬운 형태로 보여주는 것까지 포함한다.

## 24. PySpark 버전은 왜 필요한가

데이터가 커지면 pandas만으로는 처리하기 어려울 수 있다.

그래서 같은 분석을 PySpark로도 해보는 것이 중요하다.

즉, 개념은 같고 구현 환경만 달라진다.

- pandas: 소규모/학습용
- PySpark: 대용량/분산 처리용

즉, Python 실습과 PySpark 실습은  
서로 다른 것이 아니라 같은 분석을 다른 환경에서 수행하는 예시다.
