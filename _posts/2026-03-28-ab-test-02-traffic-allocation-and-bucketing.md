# A/B Test 02강 트래픽 분배 및 버킷팅 정리

## 1. 왜 트래픽을 나누는가

A/B 테스트 런타임 시스템의 가장 기본적인 역할은  
들어오는 트래픽을 실험군 A와 B로 나누는 것이다.

이 과정을 잘 설계해야:

- 같은 사용자가 일관된 경험을 하게 할 수 있고
- 실험군 간 비교가 공정해지고
- 결과 해석이 가능해진다

즉, 트래픽 분배는 단순한 구현 디테일이 아니라  
실험 신뢰도를 좌우하는 핵심 요소다.

## 2. 먼저 결정해야 하는 것: 누구를 같은 사람으로 볼 것인가

트래픽을 A/B로 나누려면 먼저  
"사용자를 식별할 기준 키를 무엇으로 잡을 것인가"를 정해야 한다.

대표적으로 많이 쓰는 기준은 다음 두 가지다.

- `user_id`
- `device_id`

이 선택은 실험의 성격에 따라 달라진다.

예를 들어:

- 로그인한 사용자만 대상으로 하는가
- 비로그인 방문자도 포함하는가
- 신규 사용자만 볼 것인가
- 기존 사용자만 볼 것인가
- 모든 방문자를 대상으로 하는가

이 질문에 따라 적절한 식별 키가 달라진다.

## 3. `user_id`란 무엇인가

`user_id`는 보통 서비스에 회원 등록이 되는 순간 부여되는 유일한 사용자 ID다.

특징:

- 한 사용자를 안정적으로 식별하기 쉽다.
- 로그인 기반 서비스에서 매우 유용하다.
- 여러 세션에 걸쳐도 같은 사람을 같은 실험군에 유지하기 쉽다.

잘 맞는 경우:

- 구독 서비스
- 로그인 후 사용이 핵심인 앱
- 추천 시스템, 결제 흐름, 계정 기능 실험

장점은 "사람" 기준으로 실험을 유지하기 쉽다는 점이다.

예:

- 오늘 로그인해서 A를 본 사용자
- 내일 다시 접속해도 계속 A를 보게 만들기 쉽다

## 4. `device_id`란 무엇인가

`device_id`는 로그인 여부와 상관없이 방문자에게 부여되는 식별자다.

웹에서는 보통 브라우저 쿠키를 이용해 만든다.

특징:

- 로그인하지 않은 사용자도 실험에 포함할 수 있다.
- 실제로는 "사용자"가 아니라 "브라우저 또는 기기"를 식별한다.
- 쿠키가 리셋되면 새로운 `device_id`가 다시 만들어질 수 있다.

즉, `device_id`는 사람 단위 식별자가 아니라  
방문 환경 단위 식별자에 가깝다.

## 5. `user_id`와 `device_id`의 차이

중요한 포인트는 둘이 1:1로 대응하지 않는다는 점이다.

- 한 `user_id`가 여러 개의 `device_id`를 가질 수 있다.
- 한 `device_id`에 여러 `user_id`가 등장할 수도 있다.

예:

- 한 사용자가 노트북, 휴대폰, 태블릿으로 접속하면 `device_id`는 여러 개다.
- 가족이 하나의 PC를 같이 쓰면 하나의 `device_id` 아래 여러 `user_id`가 생길 수 있다.

따라서 실험이 "사람 단위 경험"을 비교하는 것인지,  
"방문 단위 노출"을 비교하는 것인지에 따라 기준 키 선택이 달라져야 한다.

## 6. 어떤 상황에서 무엇을 써야 하나

### `user_id`가 더 적합한 경우

- 로그인 사용자만 대상인 실험
- 추천, 결제, 멤버십, 가격 정책처럼 사용자 수준 효과가 중요한 실험
- 여러 기기에서도 같은 사용자 경험을 유지해야 하는 실험

### `device_id`가 더 적합한 경우

- 비로그인 방문자 대상 실험
- 랜딩 페이지, 회원가입 전 퍼널, 콘텐츠 노출 테스트
- 로그인 전 구간에서 트래픽을 빠르게 나눠야 하는 실험

즉:

- "회원"을 비교하고 싶으면 `user_id`
- "방문 브라우저"를 비교하고 싶으면 `device_id`

라고 이해하면 쉽다.

## 7. `device_id` 사용 시 주의점

`device_id`는 편리하지만 몇 가지 한계가 있다.

### 7-1. 쿠키 리셋 문제

쿠키가 삭제되면 같은 사람도 새로운 `device_id`로 다시 들어온다.

그러면:

- 같은 사람이 예전에는 A를 봤는데
- 쿠키 삭제 후에는 B에 들어갈 수 있다

즉, 실험군 일관성이 깨질 수 있다.

### 7-2. 멀티 디바이스 문제

한 사람이 여러 기기에서 접속하면 기기마다 다른 실험군에 배정될 수 있다.

예:

- 모바일에서는 A
- 데스크톱에서는 B

이렇게 되면 사용자 수준 효과를 해석하기 어려워진다.

### 7-3. 봇/스크래퍼 문제

단순 크롤링이나 스크래핑 봇은 쿠키를 지원하지 않는 경우가 있다.

이 경우:

- `device_id`가 없거나
- 정상 사용자와 다른 패턴으로 관측될 수 있다

그래서 실험 로그를 해석할 때는 봇 트래픽 필터링도 중요하다.

## 8. 사용자를 나누는 방식은 크게 두 가지다

트래픽을 A/B로 나누는 방식은 크게 두 가지로 볼 수 있다.

- 미리 모든 사용자를 A/B로 나누기
- 실험 진행 중에 동적으로 나누기

## 9. 방식 1: 미리 모든 사용자를 A/B로 나누기

이 방식은 실험 시작 전에  
대상 사용자 집합을 미리 실험군 A/B로 배정해두는 방법이다.

보통 로그인 사용자 대상일 때 구현 가능하다.

예:

- 회원 테이블이 있고
- 각 사용자에게 미리 `experiment_group = A/B`를 붙여두는 방식

### 장점

- 실험 도중에 배정 기준이 흔들릴 여지가 적다.
- 여러 관점에서 bias를 줄이기 쉽다.
- 노출 전에 이미 그룹이 정해져 있어 운영 제어가 명확하다.

### 단점

- 비로그인 사용자 대상 실험에는 적용하기 어렵다.
- 실험 시작 후 새로 가입한 사용자는 별도 처리 없이는 포함되기 어렵다.
- 운영 유연성이 떨어질 수 있다.

넷플릭스에서 주로 사용하는 방식으로 소개된다.

즉, 사용자 기반이 잘 정의되어 있고  
실험 플랫폼이 성숙한 조직에서 잘 맞는 방식이다.

## 10. 방식 2: 실험 중 동적으로 나누기

이 방식은 사용자가 실제로 요청을 보냈을 때  
그 시점에 A 또는 B를 결정하는 방식이다.

이것이 일반적으로 더 널리 사용된다.

예:

1. 사용자가 서비스에 들어온다.
2. 시스템이 `user_id` 또는 `device_id`를 읽는다.
3. 버킷 계산을 한다.
4. A 또는 B variant를 보여준다.

### 장점

- 로그인 여부와 상관없이 적용 가능하다.
- 신규 방문자와 신규 가입자도 바로 실험에 포함할 수 있다.
- 운영이 유연하다.

### 단점

- 앞선 방식보다 bias가 생길 가능성이 있다.
- 여러 실험이 동시에 노출될 때 interaction 가능성을 더 주의해야 한다.

즉, 실무에서는 많이 쓰이지만  
설계를 대충 하면 실험 해석이 꼬일 수 있다.

## 11. 여기서 말하는 bias는 무엇인가

여기서 bias는  
실험군 A와 B가 원래부터 다르게 구성되어  
결과 비교가 공정하지 않게 되는 문제를 말한다.

예를 들어:

- 특정 유입 채널 사용자가 한쪽 그룹에 몰리거나
- 특정 시점 방문자가 우연히 한쪽에 더 많이 들어가거나
- 로그인 여부 변화가 실험군에 영향을 주는 경우

이런 일이 생기면  
"variant 효과"와 "집단 구성 차이"가 섞여버린다.

## 12. interaction은 왜 문제인가

interaction은 여러 실험이 서로 영향을 주는 현상이다.

예:

- 홈 화면 추천 로직 테스트
- 결제 페이지 UI 테스트

가 동시에 돌아가면 구매율 변화가:

- 추천 실험 때문인지
- 결제 실험 때문인지
- 둘의 조합 때문인지

분리해서 보기 어려워질 수 있다.

특히 동적 분배 방식에서는 사용자가 여러 실험에 동시에 들어가기 쉬워  
실험 간 간섭을 더 신경 써야 한다.

## 13. 실제 버킷팅은 어떻게 하는가

사용자를 A/B로 나누는 핵심 기술은 보통 `bucketing`이다.

흐름은 단순하게 보면 이렇다.

1. 사용자 식별값을 가져온다.
2. 그 값을 숫자처럼 다룰 수 있는 형태로 바꾼다.
3. 일정한 규칙으로 버킷 번호를 만든다.
4. 그 버킷 번호를 기준으로 A 또는 B를 결정한다.

예:

- 입력: `user_id = 12345`
- 변환: 해시 또는 숫자화
- 버킷 계산
- 결과: A 또는 B

## 14. 가장 단순한 아이디어: 나머지 연산

강의 슬라이드에서는 가장 직관적인 예로  
`modulo` 연산, 즉 나머지 연산을 보여준다.

예를 들어 어떤 ID를 숫자로 표현할 수 있다고 하면:

```python
bucket = numeric_id % 2
```

이렇게 계산할 수 있다.

그러면:

- 나머지가 `0`이면 A
- 나머지가 `1`이면 B

로 나눌 수 있다.

즉:

```python
if numeric_id % 2 == 0:
    variant = "A"
else:
    variant = "B"
```

이 아이디어의 핵심은  
"입력값을 일정 규칙으로 버킷에 넣고, 버킷에 따라 variant를 정한다"는 것이다.

## 15. 왜 실제로는 해시를 많이 쓰는가

실제 서비스에서는 ID가 단순한 연속 정수라고 가정하기 어렵다.

예:

- 문자열 형태의 `user_id`
- UUID
- 쿠키 기반 식별자

이런 값은 그대로 `modulo` 하기가 어렵다.

그래서 보통은:

1. 식별값을 해시 함수에 넣어 숫자처럼 바꾸고
2. 그 결과에 `modulo`를 적용해 버킷을 만든다

예:

```python
bucket = hash(user_id) % 100
```

그리고:

- `0~49`면 A
- `50~99`면 B

처럼 나눌 수 있다.

이 방식의 장점은:

- 같은 입력이면 항상 같은 결과가 나오고
- 빠르고
- 비교적 고르게 분산되기 쉽다는 점이다

## 16. 50:50 말고 다른 비율도 가능하다

버킷 구간을 어떻게 자르느냐에 따라  
실험 비율을 자유롭게 조정할 수 있다.

예를 들어 10%만 테스트에 넣고 싶다면:

```python
bucket = hash(user_id) % 100

if bucket < 90:
    variant = "control"
else:
    variant = "test"
```

이렇게 하면 대략:

- 90%는 control
- 10%는 test

가 된다.

실무에서는 rollout 과정에서 이런 식으로 점진적으로 비율을 높이기도 한다.

예:

- 1%
- 5%
- 10%
- 25%
- 50%

## 17. 좋은 버킷팅 로직이 가져야 하는 성질

좋은 버킷팅 로직은 보통 아래 성질을 가져야 한다.

- 결정적이어야 한다  
  같은 사용자는 다시 들어와도 같은 결과가 나와야 한다.
- 빠르게 계산 가능해야 한다  
  요청 시점에 실시간으로 처리할 수 있어야 한다.
- 충분히 고르게 분산되어야 한다  
  특정 그룹으로 치우치지 않아야 한다.
- 식별 기준이 실험 목적과 맞아야 한다  
  `user_id` 기준인지 `device_id` 기준인지가 중요하다.

## 18. 실무에서 자주 생기는 질문

### 같은 사용자가 로그인 전후로 다른 그룹에 들어가면?

이 문제는 자주 발생한다.

예:

- 로그인 전에는 `device_id` 기준으로 A
- 로그인 후에는 `user_id` 기준으로 B

이렇게 되면 한 사람이 두 variant를 경험할 수 있다.

그래서 실무에서는:

- 로그인 전후 식별자 연결 정책을 정하거나
- 특정 시점 이후에는 `user_id`를 우선하거나
- 애초에 로그인 사용자만 실험 대상으로 제한하기도 한다

즉, 식별자 전환 규칙이 매우 중요하다.

### 신규 사용자도 실험에 포함해야 하나?

실험 목적에 따라 다르다.

- 온보딩 실험이면 신규 사용자가 핵심이다.
- 리텐션/구독 실험이면 기존 사용자 중심이 더 적절할 수 있다.

중요한 것은  
"누가 모집단인가"를 먼저 정의하고 그에 맞는 식별 키와 배정 방식을 고르는 것이다.

## 19. 런타임 시스템 관점에서 보면

트래픽 분배는 런타임 시스템 안에서 대략 아래처럼 동작한다.

1. 요청이 들어온다.
2. 실험 대상 여부를 확인한다.
3. 식별자(`user_id` 또는 `device_id`)를 읽는다.
4. 버킷 계산을 한다.
5. A/B variant를 결정한다.
6. 어떤 variant가 노출됐는지 로그를 남긴다.

여기서 중요한 것은  
"노출 결정"과 "로그 기록"이 같이 가야 한다는 점이다.

그래야 나중에 분석 시스템에서:

- 누가 어떤 버전을 봤는지
- 그 뒤 어떤 행동을 했는지

를 연결해서 해석할 수 있다.

## 20. 학습용으로 꼭 기억할 핵심 문장

- A/B 테스트에서 트래픽 분배는 실험 신뢰도의 출발점이다.
- 먼저 `user_id`와 `device_id` 중 무엇을 기준으로 삼을지 정해야 한다.
- `user_id`는 사람 중심, `device_id`는 브라우저/기기 중심 식별자다.
- 사용자를 나누는 방식은 사전 배정 방식과 동적 배정 방식으로 나눠볼 수 있다.
- 실무에서는 동적 배정이 흔하지만 bias와 interaction을 더 주의해야 한다.
- 실제 버킷팅은 보통 해시 + `modulo` 방식으로 구현한다.
- 같은 사용자가 항상 같은 variant를 보게 만드는 것이 중요하다.

## 21. 한 줄 요약

A/B 테스트에서 트래픽 분배란  
실험 목적에 맞는 식별자(`user_id` 또는 `device_id`)를 기준으로  
사용자를 안정적이고 공정하게 버킷에 넣어 A와 B로 나누는 과정이다.
