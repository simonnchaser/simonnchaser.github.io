이 글은 A/B 테스트를 아이디어 수준에서 끝내지 않고 실제 서비스에서 어떻게 운영하는지에 초점을 맞춘 정리다. 실험 제안, QA, 점진적 롤아웃, 운영 UI, 분석 데이터까지 이어지는 흐름을 한 번에 묶어보면 실무에서 무엇을 준비해야 하는지가 훨씬 선명해진다.

## 1. 전체적인 A/B Test 프로세스란

A/B 테스트는 단순히 화면 두 개를 만들어 비교하는 작업이 아니다.  
실제로는 "기획 -> 승인 -> 구현 -> 검증 -> 점진적 배포 -> 분석 -> 반복 개선"까지 이어지는 하나의 운영 프로세스다.

전체 흐름은 대략 아래와 같다.

1. A/B Test Proposal & Approval
2. Implementation and QA
3. Rollout
4. Iterations
5. Periodic Review

즉, 좋은 A/B 테스트는 실험 아이디어 하나에서 끝나는 것이 아니라  
실험을 안전하게 운영하고, 결과를 해석하고, 다음 개선으로 이어지는 체계까지 포함한다.

## 2. A/B Test Proposal & Approval

실험은 보통 제안서 단계에서 시작한다. 이를 `one pager with hypothesis`라고 표현한다.

즉, 아주 길고 복잡한 문서가 아니라 핵심만 담은 1페이지짜리 정리 문서를 먼저 만드는 것이다.

보통 이런 내용이 들어간다.

- Why
- Expected Impact
- Confidence Level
- MVP to Start
- Potential Issues
- Owner

각 항목을 쉽게 풀면 다음과 같다.

### Why

왜 이 실험을 하는가를 적는다.

예:

- "현재 추천 영역 클릭률이 낮아 개인화 추천이 필요한 상황이다."
- "결제 단계가 길어 중간 이탈이 많이 발생하고 있다."

즉, 실험의 배경과 문제 정의다.

### Expected Impact

어떤 지표가 얼마나 좋아질 것으로 기대하는지 적는다.

예:

- 추천 클릭률 5% 증가 기대
- 결제 완료율 3% 증가 기대

### Confidence Level

이 가설에 대해 얼마나 확신이 있는지를 적는다.

예:

- 사용자 인터뷰와 로그 분석을 바탕으로 높은 확신
- 아이디어는 있으나 근거가 부족해 낮은 확신

이 정보는 실험 우선순위를 정할 때 중요하다.

### MVP to Start

처음부터 완벽한 기능을 다 만들지 말고,  
실험 가능한 최소 형태(MVP)가 무엇인지 정의한다.

예:

- 추천 모델 전체를 다 바꾸는 대신 홈 화면 추천 영역만 먼저 테스트
- 결제 UX 전체 개편 대신 버튼 문구와 단계 수만 먼저 테스트

### Potential Issues

실험 중 생길 수 있는 문제를 미리 적는다.

예:

- 매출 하락 가능성
- 특정 기기에서 노출 오류 가능성
- 로깅 누락 가능성

### Owner

누가 책임지고 실험을 진행하는지 명확히 해야 한다.

예:

- 구현 담당
- QA 담당
- 분석 담당

실험은 보통 여러 팀이 같이 엮이기 때문에 오너십이 중요하다.

### Stakeholder Discussion

제안서는 혼자 쓰고 끝나는 문서가 아니라,  
이해관계자들과 합의하기 위한 출발점이다.

예:

- PM은 목적과 우선순위를 확인하고
- 엔지니어는 구현 가능성을 보고
- 분석가는 측정 지표와 로그 설계를 점검하고
- 비즈니스 팀은 리스크를 확인한다

즉, 실험 전 합의가 필요하다.

## 3. Implementation and QA

실험 제안이 승인되면 실제 구현과 QA가 이어진다.

이 단계에서 중요한 것은 단순히 "기능이 돌아가느냐"가 아니라 아래 두 가지다.

- 기능 자체가 정상 동작하는가
- 실험 설정과 로깅이 정확한가

예를 들어 추천 A/B 테스트를 한다면 아래를 확인해야 한다.

- A 사용자에게는 기존 추천이 보이는가
- B 사용자에게는 새 추천이 보이는가
- 사용자 버킷 정보가 정확히 기록되는가
- 클릭, 구매, 매출 로그가 빠짐없이 남는가

즉, QA는 화면만 보는 것이 아니라 실험 데이터 품질까지 포함한다.

## 4. A/B Test Rollout Phases

실험은 보통 처음부터 50%, 100%로 크게 열지 않는다.  
점진적으로 트래픽을 늘리면서 안정성을 확인한다.

1. Smoke test
2. Initial ramp
3. Intermediate ramp
4. Final ramp-up / launch

### 4-1. Smoke test

보통 며칠 정도, 테스트 버전에 0~1%만 붙인다.

목적:

- 기능이 깨지지 않았는지 확인
- 버킷 설정이 맞는지 확인
- 로깅이 잘 되는지 확인

즉, "실험 결과를 보려는 단계"가 아니라  
"실험 자체가 안전하게 돌아가는지 확인하는 단계"다.

예:

- 결제 페이지 A/B 테스트를 0.5% 사용자에게만 먼저 노출
- 에러율, 로딩 속도, 결제 실패율을 확인

### 4-2. Initial ramp

보통 5~10% 수준까지 늘린다.

이 단계에서는:

- 시스템 이슈가 없는지
- 지표가 비정상적으로 깨지지 않는지
- 비즈니스 리스크가 감당 가능한지

를 본다.

예:

- 매출에 민감한 기능이면 5%부터 천천히
- 상대적으로 안전한 기능이면 10%까지 확대

### 4-3. Intermediate ramp

25~50% 정도까지 확장한다.

이 단계부터는 본격적으로 실험 효과를 보기 시작한다.

예:

- 추천 클릭률
- 전환율
- 매출
- 장기 리텐션

같은 지표를 비교한다.

### 4-4. Final ramp-up / launch

승자 버전이 확인되면 100%로 확대하고 최종 출시한다.

즉:

- 실험에서 성과가 좋았고
- 시스템적으로도 문제가 없고
- 리스크도 통제 가능하면

전체 서비스에 반영하는 것이다.

## 5. 왜 점진적 롤아웃이 중요한가

점진적 롤아웃은 단순히 조심스러워서 하는 것이 아니다.  
실제로 서비스 사고와 비즈니스 손실을 줄이기 위한 핵심 운영 방식이다.

예를 들어 새 추천 모델을 바로 전체 배포했는데:

- 클릭률은 올랐지만 구매율은 급락할 수 있고
- 특정 기기에서만 오류가 날 수도 있고
- 특정 국가 사용자에게만 이상 동작이 생길 수도 있다

이럴 때 100% 배포 상태라면 피해가 크다.  
반면 1%, 5%, 10%처럼 점진적으로 열면 문제를 빨리 감지하고 중단할 수 있다.

## 6. A/B Test Configuration이란

`A/B Test Configuration`은  
실험을 코드 수정 없이 운영하거나, 최소한의 코드 변경만으로 유연하게 관리할 수 있는 구조를 뜻한다.

핵심 목표는 다음과 같다.

- 코딩 없이 자주 하는 A/B 테스트를 운영할 수 있게 하기
- 반복되는 테스트 패턴을 템플릿화하기
- 운영자가 UI에서 버킷 크기나 시작/종료를 조절할 수 있게 하기

### 자주 들어가는 설정값

- Hashing parameter
- Bucket size
- Start / Stop

#### Hashing parameter

누구를 어떤 버킷에 넣을지 결정하는 기준이다.

예:

- `user_id`
- `device_id`

왜 중요하냐면, 같은 사용자가 매번 다른 버전에 들어가면 안 되기 때문이다.  
보통은 `user_id`나 `device_id`를 기준으로 해시해서 안정적으로 A 또는 B에 배정한다.

#### Bucket size

전체 트래픽 중 몇 %를 실험에 태울지 정하는 값이다.

예:

- B variant 5%
- B variant 25%
- B variant 50%

#### Start / Stop

언제 실험을 시작하고 종료할지 관리해야 한다.

이걸 코드 수정 없이 UI에서 바꾸면 운영 속도가 훨씬 빨라진다.

### 백엔드 flag 관리

실제 서비스에서는 테스트 기능을 백엔드 플래그로 관리하는 경우가 많다.

예:

- `search_v2_enabled`
- `recommendation_model_b_enabled`

이렇게 하면 특정 사용자 그룹에만 새로운 기능을 켜거나 끌 수 있다.

### 변경 이력 관리

감사 추적과 운영 기록을 위해 history는 매우 중요하다.

왜냐하면 나중에 분석할 때:

- 언제 버킷 비율이 바뀌었는지
- 누가 실험을 중지했는지
- 언제부터 언제까지 실험이 활성화됐는지

를 알아야 결과를 제대로 해석할 수 있기 때문이다.

## 7. A/B Test Admin UI는 왜 필요한가

실험을 한눈에 관리하는 Admin UI가 있으면 운영이 훨씬 수월해진다.

예를 들면 이런 정보가 보일 수 있다.

- Experiment ID
- Experiment Name
- 어느 페이지에서 도는지
- Variants
- 시작일
- 활성화 여부
- Owner
- Dashboard 링크

이런 관리 화면이 있으면:

- 어떤 실험이 현재 살아 있는지
- 누가 담당자인지
- 어디서 결과를 확인해야 하는지

를 빠르게 파악할 수 있다.

| ID  | Name      | Page   | Variants      | Starts     | Active | Owner | Dashboard |
| --- | --------- | ------ | ------------- | ---------- | ------ | ----- | --------- |
| 100 | Search V1 | Search | 200, 201      | 2017-11-01 | Yes    | Dave  | Link      |
| 101 | Reco V2   | Home   | 300, 301, 302 | 2017-11-07 | No     | Dan   | Link      |

이 표를 보면 각 컬럼이 왜 필요한지도 쉽게 이해할 수 있다.

- `ID`
  각 실험을 구분하는 고유 번호다. 로그, 대시보드, 분석 쿼리에서 실험을 정확히 식별할 때 필요하다.
- `Name`
  실험 이름이다. 예: `Search V1`, `Reco V2`
  사람이 실험 목적을 빠르게 파악할 수 있게 해준다.
- `Page`
  실험이 적용되는 서비스 영역이다. 예: `Search`, `Home`
  어떤 화면이나 기능에서 실험이 도는지 한눈에 알 수 있다.
- `Variants`
  실험에 포함된 variant ID 목록이다.
  예를 들어 `200`은 control, `201`은 test variant일 수 있다.
- `Starts`
  실험 시작일이다.
  언제부터 데이터가 쌓였는지, 분석 기간을 어디부터 잡아야 하는지 판단할 때 필요하다.
- `Active`
  현재 실험이 켜져 있는지 여부다.
  운영 중인 실험과 종료된 실험을 빠르게 구분할 수 있다.
- `Owner`
  실험 담당자다.
  이슈가 생겼을 때 누구에게 확인해야 하는지 명확해진다.
- `Dashboard`
  실험 결과를 보는 BI 대시보드나 분석 링크다.
  운영자나 분석가가 바로 결과 페이지로 이동할 수 있게 해준다.

특히 `Variants`, `Active`, `Dashboard` 컬럼은 운영 실무에서 매우 중요하다.

- `Variants`가 있어야 어떤 버전이 실험에 포함됐는지 알 수 있다.
- `Active`가 있어야 현재 켜진 실험인지 종료된 실험인지 헷갈리지 않는다.
- `Dashboard` 링크가 있어야 결과 확인과 리뷰가 빨라진다.

즉, Admin UI는 단순한 목록 화면이 아니라  
"실험 메타데이터 + 운영 상태 + 분석 접근 경로"를 한 번에 관리하는 중심 도구라고 볼 수 있다.

## 8. A/B Test 분석을 위해 필요한 데이터

실험 분석에는 크게 두 종류의 데이터가 필요하다.

1. 사용자별 버킷 정보
2. 사용자별 행동 정보

실무에서는 보통 이런 흐름으로 역할이 나뉜다.

- 백엔드/프론트엔드/앱 개발자는 실험 버킷 배정 로직과 사용자 행동 로그를 남긴다.
- 데이터 엔지니어는 이 로그들을 ETL/ELT 파이프라인으로 DW에 적재하고, 분석 가능한 형태의 테이블로 정리한다.
- 데이터 분석가는 버킷 데이터와 행동 데이터를 조인한 뒤, A/B 그룹별로 지표를 집계하고 그룹 간 통계 비교를 수행한다.

즉, A/B 테스트 분석은 단순히 분석가 혼자 하는 작업이 아니라  
개발, 데이터 엔지니어링, 분석이 함께 연결된 작업이라고 볼 수 있다.

### 8-1. 사용자별 A/B 버킷 정보

가장 먼저 알아야 할 것은 "누가 A에 들어갔고 누가 B에 들어갔는가"이다.

이 정보가 없으면 실험 분석 자체가 불가능하다.

보통 아래 같은 필드를 기록한다.

- `experiment_id`
- `variant_id`
- `timestamp`
- `user_id` 또는 `device_id`

즉, 특정 사용자가 어느 실험의 어느 variant에 속했는지를 로그로 남겨야 한다.

### 8-2. 사용자별 행동 정보

그다음 필요한 것은 사용자의 실제 행동 데이터다.

예:

- 어떤 아이템을 봤는지
- 어떤 아이템을 클릭했는지
- 어떤 아이템을 구매했는지
- 매출이 얼마였는지
- 소비/사용까지 일어났는지

이걸 흔히 `funnel data`처럼 단계별 행동 데이터로 볼 수 있다.

예:

- Impression
- Click
- Purchase
- Consumption

즉, 사용자가 퍼널의 어느 단계까지 갔는지 추적하는 것이다.

여기서 `Impression`이나 `Click to Detail Page`는 최종 핵심 지표 자체는 아닐 수 있지만,  
결과 해석과 디버깅에는 매우 중요하다.

왜냐하면 `Purchase`나 `Revenue` 같은 최종 결과만 보면  
"왜 이런 결과가 나왔는지"를 알기 어렵기 때문이다.

예를 들어 B 버전의 구매 수가 A보다 낮게 나왔다고 하자.

- 만약 `Impression`부터 크게 줄었다면:
  애초에 상품 노출이 덜 되었거나 노출 로직에 문제가 있었을 수 있다.
- `Impression`은 비슷한데 `Click to Detail Page`가 줄었다면:
  추천 카드, 제목, 썸네일, 버튼 등 관심을 유도하는 단계에서 문제가 있었을 수 있다.
- `Click to Detail Page`는 늘었는데 `Purchase`가 줄었다면:
  사용자의 관심은 끌었지만 상세페이지 이후 구매 단계에서 이탈이 커졌을 수 있다.

즉, 중간 퍼널 지표를 보면  
"문제가 노출 단계에 있는지", "클릭 유도 단계에 있는지", "구매 전환 단계에 있는지"를 나눠서 볼 수 있다.

또한 이 지표들은 실험 디버깅에도 도움이 된다.

예:

- `User Size`는 비슷한데 `Impression`만 한쪽이 크게 적다:
  일부 사용자에게 기능 노출이 안 되었거나 impression 로그가 누락되었을 수 있다.
- `Impression`은 정상인데 `Click`이 거의 없다:
  클릭 이벤트 로깅 버그, 링크 오류, UI 클릭 영역 문제일 수 있다.
- `Click`은 정상인데 `Purchase`가 급감한다:
  상세페이지 이후 결제 흐름, 재고, 가격 표시, 구매 로그 문제를 의심할 수 있다.

따라서 `Impression`, `Click to Detail Page` 같은 지표는  
최종 성과 지표를 보조하는 중간 단계 지표이자,  
실험이 어디서 잘되거나 어디서 깨졌는지를 찾는 데 중요한 단서가 된다.

## 9. 왜 Bucket Data와 Funnel Data를 JOIN해야 하는가

실험 분석의 핵심은 버킷 정보와 행동 정보를 연결하는 것이다.

쉽게 말하면:

- 이 사용자가 A였는가 B였는가
- 그리고 실제로 무엇을 했는가

를 하나로 묶어야 한다.

예를 들어:

### Bucket Data

- `experiment_id`
- `variant_id`
- `user_id`
- `timestamp`

### Funnel Data

- `user_id`
- `item_id`
- `impressions`
- `clicks`
- `revenue`

이 두 데이터를 `user_id` 기준으로 조인하면,

- A 그룹의 총 클릭 수
- B 그룹의 총 클릭 수
- A 그룹의 구매 수
- B 그룹의 매출

같은 비교가 가능해진다.

즉, 분석의 기본 구조는:

1. 실험 버킷 데이터 확보
2. 행동 로그 확보
3. 두 데이터를 조인
4. A/B로 그룹핑
5. 통계 비교

이다.

## 10. Funnel Data란 무엇인가

퍼널 데이터는 사용자가 서비스 안에서 어떤 단계들을 거쳤는지를 보여주는 데이터다.

전자상거래 예시로 보면:

- Impression: 상품이 노출됨
- Click to Detail Page: 상품 상세페이지 클릭
- Purchase: 구매
- Consumption: 실제 사용/소비

이 퍼널을 보면 단순히 "매출이 올랐나"만이 아니라,

- 노출은 많았는가
- 클릭은 잘 일어났는가
- 클릭 이후 구매까지 이어졌는가

를 단계별로 볼 수 있다.

예를 들어 B 버전이:

- Impression은 비슷한데
- Click은 높고
- Purchase는 낮다면

"흥미는 끌었지만 최종 구매까지는 못 이어졌다"는 해석이 가능하다.

## 11. A/B Test 결과 분석은 어떻게 하는가

Experiment Data와 Funnel Data를 조인하고 A와 B를 비교한다

즉, 분석은 이런 흐름으로 이루어진다.

1. 실험 데이터와 퍼널 데이터를 조인
2. A와 B로 그룹핑
3. 그룹 간 지표를 비교
4. 필요하면 사용자 메타데이터를 추가해 세부 분석

사용자 메타데이터 예:

- 연령
- 성별
- 지역
- 유입 채널
- 디바이스

이런 정보를 붙이면:

- 특정 연령대에서만 B가 잘 먹히는지
- 특정 채널 유입에서는 오히려 B가 나쁜지

같은 세분화 분석도 가능해진다.

## 12. BI Dashboard에서 무엇을 보나

BI 대시보드에서는 보통 다음 같은 지표를 본다.

- User Size
- Impressions
- Clicks
- Converted
- Revenue

그리고 필요하면 아래 차원으로 필터링한다.

- Age
- Gender
- Area
- Context
- Channel

즉, 전체 결과뿐 아니라 세그먼트별 결과도 함께 본다.

## 13. User Size와 Impressions가 왜 중요한가

### User Size

A와 B의 사용자 수는 보통 통계적으로 비슷해야 한다.

예:

- A: 50명
- B: 51명

이 정도는 자연스럽다.  
하지만 크게 차이가 난다면 실험 설정에 문제가 있을 가능성이 있다.

예:

- A: 10,000명
- B: 6,000명

이 경우는:

- 버킷 설정 오류
- 로깅 오류
- 특정 환경에서 실험 노출 누락

같은 문제를 의심해야 한다.

### Impressions

Impressions도 실험 설계상 동일해야 하는 상황이라면 비슷해야 한다.

예를 들어 새로운 기능이 "노출 수 자체"를 줄이는 기능이 아니라면:

- A와 B의 impression 수는 크게 다르지 않아야 한다

만약 많이 다르면:

- 실험이 잘못 배정되었거나
- 일부 사용자에게 기능 노출이 안 되었거나
- 로깅에 문제가 있거나
- 실험군/대조군의 트래픽 유입이 달라졌을 가능성

등을 점검해야 한다.

즉, User Size와 Impressions는 단순 참고 지표가 아니라  
실험이 정상적으로 수행됐는지 확인하는 "건강 상태 체크 지표"에 가깝다.

## 14. 쉽게 이해하는 예시

예를 들어 검색 페이지 개선 실험을 한다고 해보자.

- Experiment ID: 100
- A: 현재 검색 결과 화면
- B: 새로운 검색 결과 화면

분석 결과가 아래처럼 나왔다고 하자.

- A User Size: 10,000
- B User Size: 10,050
- A Impressions: 30,000
- B Impressions: 29,800
- A Clicks: 2,000
- B Clicks: 2,400
- A Converted: 300
- B Converted: 360
- A Revenue: 12,000
- B Revenue: 14,500

이 경우 대략 이렇게 볼 수 있다.

- 사용자 수는 비슷하므로 버킷 배정은 정상에 가까워 보인다.
- 노출 수 역시 크게 다르지 않으므로 실험 설정 문제는 커 보이지 않는다.
- 클릭, 전환, 매출은 B가 더 좋아 보인다.

즉, "새 검색 결과 화면이 실제 비즈니스 지표 개선에 도움이 되었을 가능성"이 있다.

물론 실제 의사결정에서는 통계적 유의성, 실험 기간, 세그먼트별 차이까지 함께 봐야 한다.

## 15. 전체 흐름 한 번에 정리

A/B 테스트 운영의 전체 흐름을 다시 한 줄씩 정리하면 이렇다.

1. 문제와 가설을 정리한 실험 제안서를 만든다.
2. 이해관계자와 논의하고 승인받는다.
3. 기능과 로깅을 구현하고 QA한다.
4. 0~1%부터 시작해 점진적으로 롤아웃한다.
5. 버킷 데이터와 행동 데이터를 수집한다.
6. 두 데이터를 조인해 A/B 그룹 간 지표를 비교한다.
7. 결과를 리뷰하고, 승자 버전을 출시하거나 다음 실험으로 이어간다.

## 16. 꼭 기억할 포인트

- A/B 테스트는 실험 아이디어 하나가 아니라 운영 프로세스 전체를 포함한다.
- 제안서에는 가설, 기대 효과, 리스크, 오너가 명확해야 한다.
- 실험은 보통 점진적으로 롤아웃해야 안전하다.
- 분석을 위해서는 버킷 데이터와 사용자 행동 데이터가 모두 필요하다.
- 두 데이터를 조인해야 A와 B의 성과 비교가 가능하다.
- User Size와 Impressions는 실험 이상 여부를 점검하는 중요한 기초 지표다.
