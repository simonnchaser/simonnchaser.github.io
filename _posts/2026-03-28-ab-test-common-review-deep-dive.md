# A/B Test 공통 지난강의 리뷰 딥다이브 정리

통계 검정과 귀무가설, CLT, z-test, t-test에 대한 별도 정리는 아래 문서에 이어서 정리했다.

- `notes/ab_test/A_B_Test_03강_통계_기초_및_검정_정리.md`

## 1. 왜 이 문서를 따로 정리하는가

이 문서는 A/B 테스트를 공부하면서  
한 번 더 짚고 넘어갈 만한 포인트와  
실무에서 특히 자주 헷갈리는 부분을 다시 정리한 노트다.

기본 개념을 복습하는 데서 그치지 않고:

- 왜 이 문제가 중요한지
- 어디서 실수하기 쉬운지
- 실무에서는 무엇을 추가로 봐야 하는지

를 함께 정리하는 데 목적이 있다.

## 2. A/B 테스트들 간의 Interaction이란 무엇인가

여러 A/B 테스트를 동시에 돌릴 때  
각 테스트가 서로 독립적으로 작동하지 않고 결과에 영향을 주는 현상을 `interaction`이라고 한다.

예를 들어 4명의 사용자에게 두 개의 테스트를 동시에 붙인다고 하자.

- Test 1: `A1` vs `B1`
- Test 2: `A2` vs `B2`

사용자 배정이 이렇게 되면:

- User 1 -> `A1`, `A2`
- User 2 -> `A1`, `A2`
- User 3 -> `B1`, `B2`
- User 4 -> `B1`, `B2`

실제로 테스트된 조합은:

- `A1-A2`
- `B1-B2`

뿐이다.

이 경우:

- `A1-B2`
- `B1-A2`

조합은 아예 관측되지 않는다.

즉, 두 테스트가 독립적으로 돌아간다고 볼 수 없다.

## 3. Interaction이 왜 문제인가

이 상황에서는 결과가 조합 효과인지 개별 테스트 효과인지 분리하기 어렵다.

예:

- Test 1이 좋은 것인지
- Test 2가 좋은 것인지
- `A1-A2` 조합만 특별히 좋은 것인지

를 구분하기 어려워진다.

따라서 여러 실험을 동시에 돌릴 때는  
`실험 간 독립성`이 실제로 보장되는지 항상 의심해야 한다.

## 4. Interaction을 어떻게 다루나

보통 아래 방식으로 대응한다.

- 같은 사용자 흐름에 있는 실험을 동시에 돌리지 않기
- 후처리 분석에서 실험 조합별 영향을 따로 보기
- 아예 미리 버킷을 나눠 실험끼리 간섭이 없도록 설계하기

즉, interaction은 분석 단계에서만 해결할 문제가 아니라  
버킷팅 설계 단계에서부터 고려해야 하는 문제다.

## 5. Variant 크기 비교는 왜 가장 먼저 보나

A/B 테스트를 분석할 때 가장 기본적으로 먼저 보는 것은  
Variant 간 트래픽 크기가 비슷한지 여부다.

여기서 말하는 "비슷하다"는  
완전히 똑같은 숫자라는 뜻이 아니라  
통계적으로 볼 때 오차 범위 안에 있는지를 뜻한다.

예시 표:

| Metric      | A (Control) | B (Test) |
| ----------- | ----------: | -------: |
| User Size   |          50 |       53 |
| Impressions |         120 |      115 |
| Clicks      |          10 |       15 |
| Converted   |           3 |        1 |
| Revenue     |           1 |        3 |

즉:

- A가 50명
- B가 53명

처럼 약간 차이가 나는 것은 자연스러울 수 있다.

중요한 것은  
그 차이가 단순한 무작위 변동인지,  
아니면 배정 로직이나 로그 문제를 의심해야 할 정도인지다.

## 6. 왜 이건 이항분포 관점으로 볼 수 있나

A/B 테스트에서 사용자는 보통 두 그룹 중 하나에 들어간다.

- A
- B

이 구조는 동전 던지기와 비슷하다.

- 앞면
- 뒷면

즉, 기본적으로 `binomial distribution`, 즉 이항분포 관점으로 생각할 수 있다.

그래서 Variant별 사용자 수 차이가  
자연스러운 확률 변동 범위 안에 있는지 확인하는 것이 중요하다.

실무에서는 보통 95% 신뢰구간 기준으로 많이 본다.

## 7. User Size와 Impressions를 먼저 보는 이유

아래 같은 표가 있다고 하자.

- User Size
- Impressions
- Clicks
- Converted
- Revenue

여기서 Clicks나 Revenue를 바로 해석하기 전에  
먼저 `User Size`와 `Impressions`를 본다.

왜냐하면:

- 노출 자체가 한쪽에 치우쳤는지
- 로그가 누락된 것은 아닌지
- 실험군 크기가 비정상적으로 다른지

를 먼저 확인해야 하기 때문이다.

즉, 분모가 이상하면 분자 해석도 같이 흔들린다.

## 8. A/B 테스트에서 QA는 왜 중요한가

A/B 테스트는 결국 누군가 구현해야 한다.

즉:

- 개발자 손을 타고 구현되고
- 디자이너 변경이 반영되며
- 로깅 코드가 추가되고
- 실험 설정이 붙는다

이 과정에서 원래 실험 의도가 다르게 반영될 수 있고  
단순 버그가 들어갈 가능성도 있다.

그래서 QA는 단순히 화면이 보이는지 확인하는 작업이 아니라  
실험이 의도대로 구현되고 측정되는지를 검증하는 과정이다.

## 9. QA에서 무엇을 확인해야 하나

보통 아래를 점검해야 한다.

- A 사용자는 정말 A를 보는가
- B 사용자는 정말 B를 보는가
- 버킷 로그가 정확히 남는가
- 클릭, 구매, 전환 로그가 누락되지 않는가
- 특정 기기나 브라우저에서 다르게 동작하지 않는가
- 실험 시작/종료 시점이 맞는가

즉, QA는 기능 QA와 데이터 QA를 같이 해야 한다.

## 10. 왜 A/A 테스트가 중요한가

A/A 테스트는 겉보기에는 같은 버전을 두 그룹에 나눠 보여주는 테스트다.

이 경우 원칙적으로는 큰 차이가 나오면 안 된다.

그래서 A/A 테스트는:

- 버킷팅이 정상인지
- 로그 수집이 안정적인지
- 분석 파이프라인이 이상 없는지

를 확인하는 좋은 방법이 된다.

특히 실험 시스템이 새로 만들어졌거나 변경이 있었을 때 중요하다.

## 11. 전체로는 안 좋지만 특정 세그먼트에는 좋을 수 있다

A/B 테스트 결과를 전체 평균만 보고 끝내면 중요한 신호를 놓칠 수 있다.

예를 들어:

- 전체로는 효과가 거의 없는데
- 여성 사용자에게는 매우 좋고
- 남성 사용자에게는 거의 변화가 없을 수 있다

또는:

- 고령 사용자에게는 나쁘고
- 저연령 사용자에게는 좋을 수도 있다

즉, 전체 평균은 여러 집단의 효과가 섞인 결과일 수 있다.

## 12. 세그먼트별 분석이 중요한 이유

세그먼트별 분석은 아래 질문에 답하기 위해 필요하다.

- 누구에게 효과가 있는가
- 누구에게는 오히려 해로운가
- 특정 사용자층에만 롤아웃해야 하는가

따라서 실험 결과가 애매할수록:

- 성별
- 연령
- 지역
- 유입 채널
- 신규/기존 사용자

같은 세그먼트 기준으로 다시 보는 것이 중요하다.

## 13. 지표가 꼭 좋아져야만 성공인가

항상 그렇지는 않다.

어떤 테스트는 지표가 그대로여도 성공으로 볼 수 있다.

예:

- 운영 비용이 더 적게 드는 방식으로 교체
- 레거시 시스템을 더 유지보수하기 쉬운 구조로 전환
- 장애 가능성이 더 낮은 구조로 전환

이 경우 핵심 지표가 유지되기만 해도  
교체가 충분히 가치 있을 수 있다.

즉, 성공 기준은 단순히 "모든 숫자가 올라야 한다"가 아니라  
가설과 의사결정 기준에 따라 달라질 수 있다.

## 14. 그래서 가설에 성공 기준을 미리 적어야 한다

실험 전에는 아래를 미리 정해두는 것이 좋다.

- 어떤 지표가 좋아져야 하는가
- 어느 정도까지는 같아도 괜찮은가
- 운영 비용 절감도 성공 조건에 포함되는가
- 특정 세그먼트에서만 좋아도 성공으로 볼 것인가

즉, 성공/실패 기준이 사후 해석이 아니라  
사전 가설 안에 들어 있어야 한다.

## 15. 모바일 앱 기반 A/B 테스트가 어려운 이유

모바일 앱은 웹보다 A/B 테스트가 더 까다로운 경우가 많다.

가장 큰 이유는 시간과 데이터 전송이 꼬일 수 있기 때문이다.

예:

- 사용자가 오프라인 상태에서 행동함
- 이벤트는 바로 서버로 가지 않음
- 나중에 온라인이 되었을 때 한꺼번에 전송됨

이 경우 이벤트 발생 시점과 서버 수집 시점이 달라질 수 있다.

## 16. 앱에서 시간을 그대로 믿으면 왜 위험한가

앱이 보내는 시간을 그대로 믿으면  
실험 시작 전 행동이 나중에 들어오거나,  
실험 시작 후 행동처럼 잘못 보일 수 있다.

예:

1. 사용자가 오프라인 상태에서 앱 사용
2. 실제 행동은 어제 발생
3. 오늘 온라인이 되면서 이벤트가 서버로 전송
4. 서버에서는 오늘 들어온 이벤트로 보일 수 있음

이런 상황에서는:

- 실험 시작 전/후 경계가 흐려지고
- 버킷 배정 시점과 행동 시점이 어긋나며
- 분석이 왜곡될 수 있다

즉, 모바일 앱 실험에서는 타임스탬프 정의를 매우 조심해야 한다.

## 17. 앱 업데이트가 필요한 테스트는 왜 더 느린가

웹은 서버 쪽 변경만으로 실험을 시작하기 쉬운 경우가 많다.

하지만 앱은:

- 앱 버전 배포가 필요할 수 있고
- 사용자가 실제로 업데이트해야 하며
- 충분한 설치 기반이 형성될 때까지 기다려야 한다

즉, 실험을 설계해도 바로 시작하기 어려울 수 있다.

그래서 앱 실험은:

- 최소 앱 버전 기준
- 업데이트 확산 속도
- 구버전 사용자 비율

을 함께 고려해야 한다.

## 18. 안 좋은 A/B 테스트 가설은 어떤 것인가

나쁜 가설은 보통 아래 특징을 가진다.

- 중요한 문제를 다루지 않는다
- 원인과 결과 관계가 불명확하다
- 통제하기 어려운 요소를 함께 바꾼다
- 실험으로 배우기 어려운 질문을 던진다

## 19. 잘못된 가설의 예

### 중요하지 않은 문제를 다루는 경우

예:

- 임팩트가 거의 없는 작은 기능을 굳이 큰 실험으로 검증하려는 경우

이 경우 실험 비용 대비 학습 가치가 낮을 수 있다.

### 구조적으로 잘못된 질문을 던지는 경우

예:

- CS팀 문의자의 churn rate을 낮추겠다

이건 이미 문제를 겪고 있는 사용자 집단만 보고 있어서  
해석이 꼬일 가능성이 높다.

### 컨트롤하기 어려운 가격 실험

B2C 환경에서 가격대를 바꾸는 테스트는  
수요 탄력성, 사용자 인식, 외부 공유, 형평성 이슈가 얽혀 있어 통제가 어렵다.

즉, 실험 자체는 가능해도 해석과 운영이 매우 까다롭다.

### 여러 요소를 동시에 바꾸는 경우

예:

- 검색엔진 UI와 검색 알고리즘을 동시에 바꾸는 경우

이러면 결과가 좋아져도  
무엇 때문인지 분리하기 어렵다.

그래서 가능하면 한 번에 하나의 핵심 변화만 테스트하는 것이 좋다.

## 20. A/B 테스트는 얼마나 기다려야 하나

기본적으로는 적어도 일주일은 보는 것이 안전하다.

이유:

- 요일 효과가 존재하고
- 초반 우연한 차이가 클 수 있으며
- 시간이 지나면서 유의미해 보이던 차이가 사라질 수 있기 때문이다

즉, 초반 숫자만 보고 너무 빨리 결론 내리면 위험하다.

## 21. Data Peeking Problem이란 무엇인가

`Data peeking`은 실험 결과를 너무 자주 들여다보면서  
유의미해 보이는 순간에 실험을 멈추는 문제를 말한다.

이 경우 실제로는 우연한 변동인데도  
효과가 있는 것처럼 착각할 수 있다.

예:

- 2일차에는 p-value가 유의해 보임
- 5일차에는 차이가 줄어듦
- 10일차에는 거의 차이가 없어짐

즉, 초반 신호만 보고 성급하게 결론 내리지 않는 것이 중요하다.

## 22. 최소 샘플 수는 왜 가설과 연결되어야 하나

실험을 얼마나 오래 돌릴지는  
원하는 최소 샘플 수와 연결된다.

즉, 실험 전에 아래를 정하는 것이 좋다.

- 어떤 지표를 기준으로 볼 것인가
- 최소 어느 정도 차이를 검출하고 싶은가
- 그 차이를 보기 위해 얼마의 트래픽이 필요한가

이런 기준이 없으면  
실험 종료 시점을 해석하는 사람 마음대로 정하게 될 위험이 있다.

## 23. 익숙한 UI/UX 변경은 왜 더 오래 봐야 하나

사용자가 익숙한 UI/UX가 바뀌면  
초반에는 낯설어서 성과가 떨어질 수 있다.

하지만 시간이 지나면서 적응하면  
결과가 다시 좋아질 수도 있다.

그래서 익숙한 사용 패턴을 바꾸는 테스트는  
짧게 끝내기보다 2~4주 정도 보는 것이 더 적절할 수 있다.

즉, 실험 기간은 기능의 성격에 따라 달라져야 한다.

## 24. 짧게 다시 정리

- interaction은 여러 실험이 독립적으로 작동하지 않는 문제다.
- Variant 크기 비교는 실험 해석 전에 꼭 확인해야 하는 기본 점검이다.
- QA는 기능 QA와 데이터 QA를 함께 의미한다.
- A/A 테스트는 실험 시스템과 분석 파이프라인 검증에 중요하다.
- 전체 평균만 보지 말고 세그먼트별 결과를 함께 봐야 한다.
- 지표가 같아도 운영 효율이나 유지보수성이 좋아지면 성공일 수 있다.
- 모바일 앱 실험은 시간과 이벤트 전송 문제 때문에 더 까다롭다.
- 나쁜 가설은 중요하지 않거나 통제하기 어렵거나 여러 변화를 한 번에 섞는다.
- 실험은 적어도 일주일 이상 보고, data peeking을 경계해야 한다.

실험 결과를 왜곡하거나 해석을 어렵게 만드는 실무적 변수들을 함께 봐야 한다.
