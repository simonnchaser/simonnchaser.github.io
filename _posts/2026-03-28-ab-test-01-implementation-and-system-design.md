## 1. A/B 테스트는 어떻게 구현할 수 있는가

A/B 테스트를 운영하는 방법은 크게 두 가지로 볼 수 있다.

- 직접 구현
- SaaS 사용

## 2. 직접 구현

직접 구현은 회사 내부에서 A/B 테스트 시스템을 직접 만드는 방식이다.

보통 이런 요소를 직접 만든다.

- 사용자 버킷 배정 로직
- feature flag 또는 variant 노출 로직
- 실험 설정 UI
- 버킷 로그 저장
- 사용자 행동 로그 수집
- 분석용 데이터 파이프라인
- 대시보드 또는 분석 노트북

### 장점

- 회사 서비스 구조에 맞게 자유롭게 커스터마이징할 수 있다.
- 내부 데이터와 자연스럽게 연결하기 쉽다.
- 추천 시스템, 검색 시스템, 가격 정책 등 백엔드/ML 실험에도 잘 맞는다.
- 장기적으로는 SaaS 비용이나 제약을 줄일 수 있다.

### 단점

- 초기 구축 비용이 크다.
- 운영, 유지보수, 데이터 파이프라인 관리까지 모두 내부에서 책임져야 한다.
- 실험 플랫폼 품질이 낮으면 실험 신뢰도도 떨어질 수 있다.

## 3. SaaS 사용

외부 A/B 테스트 SaaS를 사용하는 방식도 많이 쓰인다.

대표적인 서비스:

- Optimizely
- VWO

이런 도구들은 특히 Front End 관련 테스트에 유용하다.

예:

- 버튼 문구 변경
- 배너 색상 변경
- 페이지 레이아웃 변경
- 랜딩 페이지 실험

보통 Javascript를 웹사이트에 임베딩해서 동작하는 경우가 많다.

즉, 웹 페이지에 스크립트를 넣고:

- 어떤 사용자에게 A를 보여줄지 B를 보여줄지 정하고
- 전환율이나 클릭률 같은 기본 지표를 추적하는 형태다

### 장점

- 빠르게 시작할 수 있다.
- 직접 플랫폼을 만들지 않아도 된다.
- 비개발자도 비교적 쉽게 실험 설정을 다룰 수 있다.
- 프론트엔드 실험에는 진입장벽이 낮다.

### 단점

- 내부 데이터와 깊게 연결하려면 추가 연동이 필요하다.
- 추천 시스템, 검색 로직, 가격 정책, ML 모델 실험처럼 백엔드 중심 실험에는 한계가 있을 수 있다.
- 회사가 원하는 방식으로 완전히 세밀하게 제어하기 어려울 수 있다.

## 4. 실무에서는 어떻게 발전하는가

- 보통은 SaaS를 쓰다가
- 점점 직접 구현하는 식으로 고도화된다

이유는 보통 다음과 같다.

- 처음에는 빠르게 실험 문화부터 만들고 싶다.
- 이후 실험 수가 많아지고 요구사항이 복잡해진다.
- 내부 로그, 매출 데이터, 마케팅 데이터, ML 시스템과 더 깊게 연결할 필요가 생긴다.

즉, 초반에는 SaaS로 빠르게 시작하고,  
성숙한 조직은 점점 자체 플랫폼으로 확장하는 경우가 많다.

## 5. A/B 테스트 시스템은 무엇으로 이루어지는가

- 런타임 시스템
- 분석 시스템

즉:

`A/B 테스트 시스템 = 런타임 시스템 + 분석 시스템`

이 구분이 매우 중요하다.

## 6. 런타임 시스템이란 무엇인가

런타임 시스템은 실제 서비스에서 사용자를 A/B로 나누고,  
각 사용자에게 서로 다른 variant를 보여주는 시스템이다.

트래픽을 실제로 어떻게 A/B로 나누는지에 대한 별도 정리는 아래 글에 이어서 정리했다.

- [A/B Test 02강 트래픽 분배 및 버킷팅 정리]({% post_url 2026-03-28-ab-test-02-traffic-allocation-and-bucketing %})

예를 들어:

- 사용자 A는 5단계 구매 프로세스를 본다
- 사용자 B는 3단계 구매 프로세스를 본다

이 차이를 실제 서비스에서 실행하는 쪽이 런타임 시스템이다.

런타임 시스템의 핵심 역할:

- 사용자 식별
- 버킷 결정 로직 수행
- variant 노출
- 실험 상태 관리
- 버킷 정보 저장

쉽게 말하면:

- "누가 control이고 누가 test인가?"
- "이 사용자에게 어떤 화면/기능을 보여줄 것인가?"

를 실시간으로 결정하는 시스템이다.

## 7. 분석 시스템이란 무엇인가

분석 시스템은 실험 결과를 계산하고 해석하는 시스템이다.

분석 시스템에는 보통 아래 같은 데이터가 들어간다.

- A/B 테스트 사용자 버킷 정보
- Event log
- Revenue data
- Marketing data

등이 `Transform` 단계를 거쳐 대시보드로 올라간다.

즉 분석 시스템의 핵심 역할은:

- 버킷 데이터 수집
- 행동 로그 수집
- 매출/마케팅/기타 메타데이터 결합
- ETL/ELT 또는 Transform 수행
- A/B 비교 지표 계산
- 대시보드 제공

쉽게 말하면:

- "A와 B 중 누가 더 좋았는가?"
- "어느 단계에서 차이가 났는가?"
- "세그먼트별로 결과가 어떻게 다른가?"

를 답해주는 시스템이다.

## 8. 그림으로 이해하는 시스템 흐름

1. 사용자가 서비스에 들어온다.
2. 버킷 결정 로직이 사용자를 A 또는 B에 배정한다.
3. 각 사용자에게 다른 variant가 노출된다.
4. 이때 버킷 정보가 저장된다.
5. 사용자의 행동 로그, 매출 데이터, 마케팅 데이터 등이 계속 쌓인다.
6. 이 데이터들이 Transform 과정을 거쳐 분석용 테이블로 정리된다.
7. 최종적으로 대시보드에서 A/B 결과를 비교한다.

즉, A/B 테스트는 단순히 화면 두 개를 보여주는 기능이 아니라  
"실시간 노출 시스템"과 "사후 분석 시스템"이 연결되어야 완성된다.

## 9. 전체 과정 다시 보기

전체 A/B 테스트 과정은 보통 아래 순서로 진행된다.

1. A/B Test Proposal & Approval
2. Discussion with stakeholders
3. Implementation and QA
4. Rollout
5. Iterations

이를 조금 더 풀면:

### 9-1. Proposal & Approval

- one pager with hypothesis 작성
- 왜 이 실험을 하는지 정리
- 기대 효과와 지표 정의
- 이해관계자와 논의

### 9-2. Implementation and QA

- 기능 구현
- 버킷 배정 로직 점검
- 로그 누락 여부 확인
- 실험 설정 검증

### 9-3. Rollout

- 작은 비율부터 시작
- 문제 없으면 테스트 퍼센트 증가

예:

- 1%
- 5%
- 10%
- 25%
- 50%
- 75%
- 100%

### 9-4. Iterations

- 주간 리뷰
- 대시보드 확인
- 결과 해석
- 다음 실험 아이디어 반영

즉, 실험은 한 번 하고 끝나는 것이 아니라 반복 개선 구조를 가진다.

## 10. 대시보드가 왜 중요한가

대시보드가 있으면:

- 실험 결과를 빠르게 확인할 수 있다.
- 팀이 같은 숫자를 보며 논의할 수 있다.
- 세그먼트별 분석이 쉬워진다.
- 주간 리뷰 속도가 빨라진다.

예시 도구:

- Tableau
- Looker
- Power BI
- Python Notebook

결국 중요한 것은 특정 툴 이름보다,  
"실험 결과를 빠르게 확인하고 반복할 수 있는 분석 환경이 있는가"이다.

## 11. 왜 여기서부터 Agile A/B Test가 중요한가

한 번의 실험에서 끝나는 경우는 드물다.  
보통은 아래 반복이 계속 일어난다.

- 결과 분석
- 알고리즘/설정 수정
- 다시 배포
- 다시 데이터 수집
- 다시 분석

즉, 중요한 것은 단순히 실험을 할 수 있느냐가 아니라  
"이 반복을 얼마나 빠르게 돌릴 수 있느냐"이다.

## 12. Iteration을 빠르게 한다는 것은 무슨 뜻인가

- algorithm / config update
- algorithm / config deployment
- experiment setup
- data collection
- analysis

그리고 다시 update로 돌아간다.

즉, A/B 테스트의 반복 사이클은 대략 이런 구조다.

1. 알고리즘 또는 설정을 바꾼다.
2. 새 버전을 배포한다.
3. 실험을 설정한다.
4. 데이터를 수집한다.
5. 결과를 분석한다.
6. 분석 결과를 바탕으로 다시 수정한다.

이 한 바퀴가 빠를수록:

- 실험을 더 많이 할 수 있고
- 더 빨리 배울 수 있고
- 더 빨리 좋은 버전을 찾을 수 있다

- ideal cycling time is days

즉, 이상적으로는 몇 달이 아니라 며칠 단위의 반복 속도가 좋다는 뜻이다.

## 13. 왜 agility가 이기는가

`agility wins the quarter`는 결국 이런 의미다.

- 더 빨리 실험하는 팀이
- 더 빨리 실패하고
- 더 빨리 배우고
- 더 빨리 좋은 버전을 출시한다

즉, A/B 테스트의 경쟁력은 통계 지식만이 아니라  
실험 시스템과 분석 시스템의 반복 속도에도 달려 있다.

## 14. 짧게 정리

- A/B 테스트 구현 방식은 `직접 구현`과 `SaaS 사용`으로 나눌 수 있다.
- SaaS는 빠르게 시작하기 좋고, 직접 구현은 복잡한 내부 실험에 유리하다.
- 실무에서는 SaaS로 시작한 뒤 직접 구현으로 고도화되는 경우가 많다.
- A/B 테스트 시스템은 `런타임 시스템`과 `분석 시스템`으로 구성된다.
- 실험 운영의 핵심은 `제안 -> 구현 -> 롤아웃 -> 반복 개선`이다.
- 좋은 대시보드는 반복 속도를 높여준다.
- 결국 Agile A/B Test의 핵심은 실험 iteration을 빠르게 돌리는 것이다.

## 15. 한 줄 요약

A/B 테스트를 잘하려면 단순히 실험 아이디어만 있는 것이 아니라,  
사용자를 실시간으로 버킷에 배정하는 런타임 시스템과 결과를 빠르게 해석하는 분석 시스템이 함께 있어야 하며,  
이 전체 반복 사이클이 빠를수록 더 좋은 제품 개선이 가능하다.
